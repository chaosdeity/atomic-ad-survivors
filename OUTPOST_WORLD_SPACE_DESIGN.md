# Outpost World-Space Design

작성일: 2026-05-19
범위: 침묵 보급소를 실제 RPG 허브 공간으로 확장하기 위한 설계
비범위: 코드 구현, Godot import, PNG 생성/교체, 저장/불러오기, NPC 구현, 성장/보상/전투 수치 변경

## 1. 역할 정의

침묵 보급소는 업글 메뉴가 아니다. 이곳은 윤서가 캠페인 환경에서 회수된 뒤 다시 사람의 이름과 물건의 용도를 확인하는 작은 거점이다.

보급소의 핵심 역할:

- 로그라이크 런 사이의 허브.
- RPG 성장, 정산, 보관, 선택, 재출격을 묶는 공간.
- HP 0이나 신호 과부하가 "죽음"이 아니라 "회수 후 재정비"로 읽히게 하는 장소.
- 플레이어가 점점 강해지는 이유를 시설, 흔적, 문구, 장비 변화로 보여주는 장소.
- 전단, 코어 파편, 문패, 가족사진, 영수증, 송출 기록이 숫자만이 아니라 물리적으로 남는 장소.
- 0.3 이후 NPC, 시설, 작은 맵 이동, 지역 반응, 후속 출격 선택을 받을 수 있는 RPG 허브.

현재 구현은 `scripts/hud_controller.gd`의 `supply_panel` 위에 visual placeholder와 16개 업글 목록이 뜨는 구조다. 이 구조는 0.2에서 유지한다. 다만 설계상 이 UI는 최종 형태가 아니라, world-space 보급소 안의 시설을 임시로 한 화면에 압축한 상태로 본다.

핵심 문장:

```text
보급소는 구매 화면이 아니라, 윤서가 돌아와서 다시 나갈 이유를 정리하는 장소다.
```

## 2. World-Space 크기 기준

`480x270`은 카메라 한 화면과 UI/mock 가독성 기준이지, 보급소 실제 크기가 아니다. 보급소는 작은 RPG 허브이므로 world-space 기준으로 확장 가능해야 한다.

크기 후보:

| 단계 | 가정 크기 | 목적 |
| --- | ---: | --- |
| 최소 테스트 | 960x540 | 2x2 camera screen. 회수 플랫폼, 정산 카운터, 정비대, 게시판, 재출격 지점을 한 번에 검증 |
| 권장 0.2 확장 기준 | 1440x810 | 3x3 camera screen. 시설 간 간격, 중앙 동선, 상태 변화 prop을 분리 |
| 확장 가능 기준 | 1920x1080 이상 | NPC, 닫힌 통로, 흔적 보관실, 지역 선택 wing, 후속 챕터 시설 추가 |

최종 크기는 지금 확정하지 않는다. 0.2에서는 UI 패널 배경을 강화하거나 코드 드로잉 blockout을 붙이고, 0.3 이후 별도 `outpost_world` scene 또는 hub controller로 이동할 수 있게 설계한다.

권장 공간 구성:

```text
top / north:
  출격 게시판, 닫힌 통로, 상위 지역 gate 후보

center:
  회수 플랫폼, 넓은 clear path, 윤서 시작/귀환 지점

west:
  정산 카운터, 흔적 보관실, 재화/기록 확인

east:
  정비대, 차징 조율대, 장비/업글 선택

south:
  재출격 게이트, 회수선 릴, 다음 sortie 진입
```

## 3. 하위 구역

### 3.1 회수 플랫폼

위치 논리:

- 보급소 중앙 또는 남쪽 중앙.
- 윤서가 런 종료 후 처음 나타나는 곳.
- 다른 시설로 이동하기 전 반드시 시야에 들어와야 한다.

플레이어가 가는 이유:

- 회수 직후 상태를 확인한다.
- 재출격 게이트와 가장 가까운 기준점이다.
- 첫 강제 회수가 "죽음"이 아니라 보급소가 당긴 회수선이라는 사실을 납득한다.

붙는 UI/상호작용:

- 결과 화면 이후 보급소 진입 위치.
- 재출격 버튼의 물리 anchor.
- 0.3 이후 `exit_trigger` 또는 출격 확인 상호작용.

필요 시각 요소:

- 바닥에 회수 원형/릴 흔적.
- 끊어진 광고 노이즈가 플랫폼 경계에서 조용해지는 표현.
- 윤서 이름표/회수선이 잠시 안정되는 작은 표시.

0.2 최소 구현:

- 현재 `회수선 고정 흔적` placeholder를 보급소 중앙 anchor로 유지.
- 결과 화면에서 보급소로 넘어올 때 이 플랫폼이 배경에서 읽히게 한다.
- 재출격 버튼은 기존 UI 방식을 유지한다.

나중 확장:

- 회수 직후 짧은 착지/노이즈 감소 연출.
- 캐릭터별 회수 반응.
- 출격 지역 선택 gate와 연결.

### 3.2 정산 카운터

위치 논리:

- 회수 플랫폼에서 왼쪽 또는 하단 왼쪽.
- 귀환 직후 가장 먼저 들르게 되는 시설.
- 출격 게시판과 너무 붙이지 않아, 정산과 다음 목표가 구분되어야 한다.

플레이어가 가는 이유:

- 생존 시간, 처치, 전단, 코어 파편, 송출 기록, 흔적 회수 결과를 확인한다.
- "성과 때문에 보상을 받았다"는 원인을 확인한다.

붙는 UI/상호작용:

- 결과 정산 UI.
- 재화/흔적 summary.
- 첫 강제 회수, 정상 회수, 보스 회수, 보스 처리 결과의 문구 분기.

필요 시각 요소:

- 영수증 프린터.
- 전단 정리함.
- 코어 파편 tray.
- 회수 결과가 인쇄되는 종이/스탬프.

0.2 최소 구현:

- 현재 결과 화면과 보급소 `supply_label`의 재화 문구를 정산 카운터 anchor로 본다.
- `찢어진 광고 전단`, `캠페인 코어 파편`이 카운터 위의 물건처럼 보이게 배경 placeholder를 유지/강화할 수 있다.

나중 확장:

- 정산 카운터 상호작용으로 result log 다시 보기.
- 자동 정산과 중요 흔적 선택 분리.
- 정산 담당 NPC 또는 기계 추가.

### 3.3 정비대

위치 논리:

- 보급소 오른쪽 또는 동쪽 중단.
- 차징 조율대와 가깝지만 완전히 겹치지 않는다.
- 회수 플랫폼에서 짧게 걸어갈 수 있어야 한다.

플레이어가 가는 이유:

- 반품 스캐너, 자동공격, 기본 생존 업글을 구매한다.
- 전단 조각이 장비 절연재/보정판으로 바뀌는 이유를 본다.

붙는 UI/상호작용:

- 기존 16개 업글 UI 중 자동공격, 사거리, 엘리트 대응, 일부 생존 계열.
- `광고 잔상 해독`, `자동 조준 렌즈`, `엘리트 광고 해체`, `보급소 응급처치` 같은 항목의 공간 anchor.

필요 시각 요소:

- 반품 스캐너 수리대.
- 공구, 작은 조명, 부품 tray.
- 업글 구매 후 켜지는 조명/라벨.

0.2 최소 구현:

- 기존 업글 목록은 유지한다.
- 버튼 텍스트와 보급소 배경에서 정비대가 "구매 목록"의 물리 출처로 읽히게 한다.

나중 확장:

- 정비대 앞에서 상호작용하면 관련 업글만 필터링.
- 윤서 장비 외형 변화 preview.
- 장도윤 같은 정비 NPC 배치.

### 3.4 이름 보관함

위치 논리:

- 조용한 벽면, 오른쪽 상단 또는 북동쪽.
- 전투 장비 시설과 살짝 떨어져 있어야 한다.
- 보급소의 감정적 중심이다.

플레이어가 가는 이유:

- 지워지지 않은 문패, 반쯤 탄 가족사진, 이름 없는 영수증 같은 흔적이 사라지지 않았음을 확인한다.
- 보스 분석과 처리 결과가 장소에 남았는지 확인한다.

붙는 UI/상호작용:

- 흔적 보관 summary.
- 보스 분석 1/2/3.
- `destroy_node` / `extract_memory` 결과 hook.
- 0.3 이후 흔적 보관/소모/캠페인 활용 선택.

필요 시각 요소:

- 문패 슬롯.
- 사진/영수증 보관 칸.
- 코어 파편 유리병.
- 잠긴 칸과 빈 라벨.

0.2 최소 구현:

- 현재 `이름 보관함` placeholder를 유지한다.
- 보스 HP 65/25 분석, 결절 파괴, 기억 추출 문구가 이 시설에 연결된다고 문서/문구상 명시한다.

나중 확장:

- 흔적별 slot visual.
- 선택에 따라 보관함 색/조명 변화.
- 윤서 이름표 또는 맡아둔 이름표의 장기 서사 연결.

### 3.5 출격 게시판

위치 논리:

- 보급소 북쪽 또는 상단 오른쪽.
- 회수 플랫폼에서 바로 보이지만, 정산 카운터와 분리.
- 재출격 게이트와 시선이 이어져야 한다.

플레이어가 가는 이유:

- 다음 목표를 확인한다.
- 송출 기록 1/2/3 진행도를 이해한다.
- 모델하우스 결절, 스마일 홈 시어머니, 후속 송출관 잔향을 따라갈 이유를 얻는다.

붙는 UI/상호작용:

- `next_objective`, `route_stage_label`, `boss_signal_label`.
- R01 재방문 목표.
- 0.3 이후 지역/미션 선택 UI.

필요 시각 요소:

- 찢긴 기록 1/2/3.
- open-house 방향 핀.
- 실핀, 실선, 지도 조각.
- 보스 처리 후 끊긴 route 또는 사진 잔향 route.

0.2 최소 구현:

- 현재 `supply_label`의 `게시판`, `다음 목표` 문구를 출격 게시판 anchor로 유지한다.
- 송출 기록 수에 따라 게시판 문구가 달라지는 구조를 보급소 상태 변화 기준으로 잡는다.

나중 확장:

- 실제 게시판 상호작용.
- R01 / 후속 지역 / 송출관 잔향 선택.
- 미션 유형 선택.

### 3.6 차징 조율대

위치 논리:

- 정비대 옆 또는 남동쪽.
- 반품 스캐너 정비와 구분되는 작은 장치 공간.
- 재출격 전 마지막으로 들르는 장비 점검 위치가 될 수 있다.

플레이어가 가는 이유:

- 반품 도장, 회수선 안정화, 보스 코어 노출 관련 업글을 이해한다.
- 차징 손맛과 보스 대응이 성장한다는 이유를 본다.

붙는 UI/상호작용:

- 차징 피해, 차징 입력 윈도우, 차징 쿨다운, 코어 신호 판독, 방패 주파수 픽업, 좌표 연동 해석.
- 회수선 안정화 계열.

필요 시각 요소:

- 도장 압력 게이지.
- 메트로놈/타이밍 링.
- 회수선 릴과 케이블.
- 코어 파편을 꽂는 작은 socket.

0.2 최소 구현:

- 기존 업글 목록 안에서 차징/보스 대응 업글이 이 시설과 연결된다고 명시한다.
- 보스 HP 65/25 분석 후 조율대에 코어 파편/노이즈 meter가 생기는 방향으로 잡는다.

나중 확장:

- 차징 연습/테스트 dummy.
- 보스 패턴 힌트 replay.
- 캐릭터별 차징 스킬 교체/조율.

### 3.7 흔적 보관실

위치 논리:

- 이름 보관함 뒤쪽 또는 서쪽 카운터 뒤쪽.
- 0.2에서는 닫힌 문/잠긴 선반으로만 암시 가능.

플레이어가 가는 이유:

- 보관한 흔적을 다시 보고, 흔적 선택 결과를 확인한다.
- 장기적으로 인간/로봇/캠페인 관련 큰 선택을 관리한다.

붙는 UI/상호작용:

- 흔적 상세 보기.
- 보관/소모/캠페인 활용 선택.
- NPC 반응 archive.

필요 시각 요소:

- 닫힌 선반.
- 분류 태그.
- 작은 보관 상자.
- 손글씨 라벨.

0.2 최소 구현:

- 실제 방 구현 금지.
- 이름 보관함 옆 잠긴 칸 또는 문구로만 암시.

나중 확장:

- 흔적별 inspect scene.
- 장기 mystery layer와 연결.
- 보스 outcome별 archive 변화.

### 3.8 닫힌 통로 / 아직 열리지 않은 구역

위치 논리:

- 북쪽 또는 동쪽 끝.
- 출격 게시판 뒤쪽, 혹은 보급소 깊은 곳.
- 현재 기능과 직접 연결되지 않는 확장 여백.

플레이어가 가는 이유:

- 지금은 갈 수 없지만 보급소가 더 큰 장소라는 인상을 받는다.
- 후속 지역, NPC, 로봇 시설, 송출관 연구실이 붙을 수 있다는 기대를 남긴다.

붙는 UI/상호작용:

- 0.2에서는 없음.
- 0.3 이후 지역 해금, NPC 방, 라디오실, 기억 보관실, 로봇 충전실 후보.

필요 시각 요소:

- 반쯤 내려간 셔터.
- `아직 침묵권 불안정` 같은 자연문 경고.
- 잠긴 문 뒤 희미한 장비 silhouette.

0.2 최소 구현:

- 배경 prop만 가능.
- 버튼/상호작용처럼 보이지 않게 한다.

나중 확장:

- 라디오실.
- NPC 숙소/정비실.
- 지역 선택 corridor.
- 캠페인 송출관 연구 wing.

## 4. 현재 UI와 공간의 관계

중요 원칙:

- 당장 UI를 없애지 않는다.
- 0.2에서는 UI 유지 + 배경 공간 강화가 맞다.
- 0.3 이후에는 실제 이동/상호작용으로 쪼갤 수 있게 UI anchor를 시설별로 고정한다.

현재 UI 연결:

| 현재 UI/데이터 | 공간 anchor | 장기 연결 |
| --- | --- | --- |
| 결과 화면 | 정산 카운터 | 카운터 상호작용으로 최근 정산 log 확인 |
| `trace_torn_ad_flyer`, `campaign_core_fragment` | 정산 카운터 + 이름 보관함 | 재화는 카운터, 코어 파편/보스 분석은 보관함 |
| 16개 업글 목록 | 정비대 + 차징 조율대 | 0.3에서 카테고리별 시설 필터 |
| 자동공격/사거리/엘리트 대응 | 정비대 | 반품 스캐너 정비 |
| 차징/회수선/보스 대응 | 차징 조율대 | 반품 도장, 회수선, 코어 판독 |
| 보스 분석 1/2/3 | 이름 보관함 + 차징 조율대 | 기록 벽과 코어 판독 장치 |
| 송출 기록 1/2/3 | 출격 게시판 | 게시판 핀/지도/기록 카드 |
| 다음 목표 | 출격 게시판 | 지역/미션 선택 |
| 재출격 버튼 | 회수 플랫폼 / 출격 게이트 | 플랫폼 앞 exit trigger |
| `destroy_node` | 이름 보관함 + 출격 게시판 | 조용해진 문패 slot, 끊긴 route |
| `extract_memory` | 이름 보관함 + 흔적 보관실 | 사진/영수증 잔향, 후속 기억 route |

업글 목록 분리 후보:

| 계열 | 시설 |
| --- | --- |
| 기본 전투 | 정비대 |
| 차징 | 차징 조율대 |
| 생존/회수선 | 회수 플랫폼 + 차징 조율대 |
| 보스 대응 | 이름 보관함 + 차징 조율대 |
| 신호 분석 | 출격 게시판 + 정산 카운터 |
| 성장/XP | 정산 카운터 + 정비대 |

0.2에서는 이 분리를 UI로 강제하지 않는다. 대신 문구와 배경 anchor만 맞춘다.

## 6. Collision / Navigation 기준

보급소도 실제 맵이 될 수 있으므로 R01과 같은 분류를 사용하되, 전투 맵보다 동선 안정성이 더 중요하다.

### 6.1 분류

| 분류 | 의미 | 보급소 후보 |
| --- | --- | --- |
| `hard_blocker` | 플레이어/NPC가 통과하지 못하는 실제 구조 | 벽, 큰 카운터 뒤, 잠긴 문, 큰 보관함, 발전기 |
| `soft_blocker` | 약한 우회/시각 밀도. 통과 가능하거나 낮은 비용 | 의자, 작은 상자, 케이블 더미, 낮은 전단 박스 |
| `interaction_anchor` | 상호작용 지점. 충돌보다 위치/프롬프트가 중요 | 정산 카운터 앞, 정비대 앞, 게시판 앞, 보관함 앞, 출격 게이트 |
| `decorative` | 충돌 없음. 상태 표현/분위기 | 전단 조각, 라벨, 벽 note, 작은 조명, 바닥 선 |
| `exit_trigger` | 다른 상태/출격으로 전환 | 재출격 게이트, 회수 플랫폼 edge, 나중 지역 이동 문 |

### 6.2 동선 규칙

- 회수 플랫폼에서 정산 카운터, 정비대, 출격 게시판, 재출격 게이트까지 막힘 없는 1차 동선을 둔다.
- 플레이어가 시설 뒤에 끼면 안 된다.
- 재출격 지점은 항상 명확해야 한다.
- 업글/정산/게시판 상호작용 지점은 서로 너무 붙으면 안 된다.
- 0.2 권장 기준에서 각 주요 `interaction_anchor` 사이에는 최소 한 캐릭터 폭의 4~6배 이상 여백을 둔다.
- NPC가 추가되어도 중앙 clear path를 막지 않게 NPC idle 위치는 시설 옆이나 벽면에 둔다.
- 닫힌 통로는 `hard_blocker`지만, 실제 버튼처럼 보이면 안 된다.
- 장식 케이블/전단은 `decorative` 또는 `soft_blocker`로 두고 출격 게이트 앞을 막지 않는다.

### 6.3 구역별 충돌 의도

| 구역 | Hard | Soft | Interaction | Exit |
| --- | --- | --- | --- | --- |
| 회수 플랫폼 | 낮음 | 낮음 | 중앙 회수 anchor | 재출격 게이트와 연결 |
| 정산 카운터 | 카운터 뒤 | 전단 상자 | 카운터 앞 | 없음 |
| 정비대 | 작업대 body | 공구/케이블 | 작업대 앞 | 없음 |
| 이름 보관함 | 보관함 body | 작은 상자 | 슬롯 앞 | 없음 |
| 출격 게시판 | 벽/게시판 body | 핀/상자 | 게시판 앞 | 0.3 지역 선택 가능 |
| 차징 조율대 | 조율 장치 body | 케이블 | 조율대 앞 | 없음 |
| 닫힌 통로 | 셔터/문 | 경고 표지 | 없음 또는 잠김 prompt | 0.3 이후 |

## 7. 0.2 최소 구현 범위

0.2에서 필요한 최소 표현:

- 보급소 배경 공간 1장 또는 코드 드로잉 blockout.
- 시설 5개 이상 placeholder: 회수 플랫폼, 정산 카운터, 정비대, 이름 보관함, 출격 게시판, 차징 조율대 중 최소 5개.
- 현재 `supply_panel`, 결과 화면, 16개 업글 UI 유지.
- 보급소 진입 시 공간 배경이 보이고, UI가 그 위에 얹혀 있는 느낌.
- 업글/재출격 버튼은 기존 방식 유지.
- 상태별 게시판 문구 최소 변경: 첫 회수, 송출 기록 1/2/3, 보스 분석, 보스 처리, `destroy_node`, `extract_memory`.
- 내부 수치 대신 자연문으로 표현.
- 디버그 id, collision type, campaign pressure 수치 일반 화면 노출 금지.

0.2에서 하지 않는 것:

- 실제 보급소 이동 구현.
- 실제 NPC 이동/상호작용 구현.
- 장문 NPC 대화창 구현.
- 별도 facility UI 분리.
- 저장/불러오기.
- 장비/상점/가챠.
- 새로운 업글 수치.

## 12. NPC 로그/표식 1차 구현 기록

### 12.1 구현 반영 파일

- `scripts/meta_progression.gd`
- `scripts/main.gd`
- `scripts/hud_controller.gd`
- `scripts/outpost_layout_blockout.gd`

### 12.2 범위

이번 구현은 풀 NPC 시스템이 아니다.

0.2 범위는 다음까지만 둔다.

- 보급소 핵심 NPC 5명의 미니맵 표식
- 최근 반응 NPC 강조
- 정산/오염/배분/강화/신호/보스 결과에 따른 짧은 이벤트 로그
- 보급소 화면의 `보급소 로그:` 표시

아직 하지 않는다.

- NPC 이동
- NPC 클릭 상호작용
- 장문 대화창
- 호감도/개인 퀘스트
- 별도 NPC 메뉴

| 도윤 | 정비대/차징 조율대 | 정비 강화, 보스 분석, 결절음 변화 |

### 12.4 의미

보급소는 이제 다음처럼 읽힌다.

```text
정산 결과
-> NPC 한 줄 반응
-> 최근 로그 표시
-> 미니맵 NPC 표식 강조
-> 재출격 목표 확인
```

1. NPC 표식이 보급소 공간감을 강화하는가?
2. 로그가 짧고 조건에 맞게 읽히는가?
3. 보급소 화면이 과금/상점 화면이 아니라 귀환 장소로 느껴지는가?
