# Godot Scope 0.2 From Story

상태: 0.2 Godot 구현 범위 확정 문서  
주의: 이 문서는 코드 구현이 아니라, 코드 작업으로 넘길 스토리/데이터 범위 고정 문서다.  
연결 문서: `story/90_implementation/implementation_0_2_story_directive.md`, `story/90_implementation/reward_growth_loop_0_2.md`, `story/90_implementation/outpost_growth_minimum_0_2.md`, `story/90_implementation/r01_suburb_content_lock_0_2.md`, `story/90_implementation/character_score_upgrade_backlog.md`

## 목적

0.2에서 Godot 코드로 옮길 스토리 범위를 확정한다.

기준:

```text
작게 만들되, Atomic Ad Survivors의 핵심 재미가 빠지면 안 된다.
```

0.2는 전체 게임이 아니다. 하지만 아래 네 가지는 반드시 증명해야 한다.

- 윤서가 기본 캐릭터로 좋아질 가능성이 있다.
- 외곽 주택가는 한 번 쓰고 버리는 맵이 아니다.
- 보상은 죽음이 아니라 런 성과에서 나온다.
- 보급소는 메뉴가 아니라 돌아오는 장소다.

## 0.2 구현 패키지

### P0: 필수

이것이 없으면 0.2의 스토리 구조가 성립하지 않는다.

| 패키지 | 내용 |
| --- | --- |
| 기본 캐릭터 | 윤서, 반품 스캐너, 반품 도장 |
| 첫 강제 회수 | 100초 전후 `signal_overload` |
| 반복 재화 | 광고 전단 조각 |
| 보스 재화 | 캠페인 코어 파편 |
| 스토리 단서 | 송출 기록 3개 |
| 첫 지역 | 외곽 주택가 |
| 적 3종 | 쿠폰 전단 떼, 미소 우편함, 세일 풍선 |
| 엘리트 | 행복 보증 상담원 |
| 첫 보스 | 스마일 홈 시어머니 |
| 보스 선택 | 결절 파괴, 기억 추출 |
| 보급소 | 정산, 정비, 출격 게시판, 이름 보관함 |
| 흔적 3개 | 문패, 가족사진, 이름 없는 영수증 |
| 지역 상태 | `campaign_pressure`, `boss_outcome`, `revisit_count`, `adaptation_charge` |

### P1: 가능하면 포함

0.2 완성도를 크게 올리지만, 일정이 위험하면 줄일 수 있다.

| 패키지 | 내용 |
| --- | --- |
| 적 1종 추가 | 홈케어 청소기 |
| 흔적 3개 추가 | 고장 난 마스코트 배지, 무음 라디오, 납 차폐 배터리 |
| 보급소 변화 | 개조 후 오브젝트 변화 |
| 재방문 변형 | 단골 고객 주간, 침묵 배수로 |
| 윤서 추가 반응 | 흔적 4개 반응, 반복 방문 대사 |
| 차징 적응 | 가족 플래너 엘리트 힌트 |

### P2: 보류

문서에는 남기지만 0.2 코드 범위에서 제외한다.

| 보류 | 이유 |
| --- | --- |
| 팝시-제로 플레이어블 | 0.3 이후 수익/확장 후보 |
| 나리 플레이어블 | 흔적 시스템 안정 후 |
| 고위험 역송출형 | 캠페인 활용 시스템 안정 후 |
| 코어 해방 선택 | 로봇 감정선 필요 |
| 신호 우회 선택 | 위험 보상 루트가 아직 이르다 |
| 캠페인 송출관 보스전 | 상위 접근 목표로만 둔다 |
| 진짜 악의 행위자 등장 | 0.2에서는 씨앗만 |
| 최종 엔딩 | 장기 구조 |

## 런 결과 처리

런 종료 후 순서:

```text
1. 생존/회수/인양/보스 처리 판정
2. 성과 기반 보상 계산
3. 지역 상태 갱신
4. 행동 기억 갱신
5. 송출 기록/흔적/보스 선택 처리
6. 보급소 정산 화면
7. NPC 반응 1~2개
8. 출격 게시판 갱신
```

중요:

- 회수/인양은 보상 원인이 아니다.
- 보상 원인은 생존 시간, 처치 수, 엘리트 처치, 신호 이벤트, 보스 피해, 흔적 회수다.
- 첫 강제 회수는 실패가 아니라 `signal_overload` 이벤트다.

## 첫 10분 필수 흐름

```text
게임 시작
-> 윤서로 외곽 주택가 출격
-> 자동공격 학습
-> 차징 반품 도장 학습
-> 캠페인이 윤서 이름을 읽으려 함
-> signal_overload 강제 회수
-> 침묵 보급소 귀환
-> 광고 전단 조각 정산
-> 첫 개조 1개 가능
-> 재출격 유도
```

합격 감정:

```text
죽은 게 아니라 돌아왔고, 다음엔 더 멀리 갈 수 있겠다.
```

## 첫 1시간 필수 흐름

```text
송출 기록 1 확보
-> 보급소 해석
-> 송출 기록 2 확보
-> 송출 기록 3 확보
-> 송출관 접근 절차 확정
-> 스마일 홈 시어머니 조우
-> 보스 HP 구간 성과 또는 보스 처리
-> 결절 파괴/기억 추출 선택
-> 외곽 주택가 게시판 변화
```

합격 감정:

```text
같은 지역인데, 내가 한 선택 때문에 다음 런의 공기가 바뀌었다.
```

## 정산 화면 최소 정보

필수:

- 생존 시간.
- 처치 수.
- 광고 전단 조각.
- 캠페인 코어 파편.
- 송출 기록 획득 여부.
- 흔적 획득 여부.
- 추천 개조 1개.
- NPC 반응 1개.

예시:

```text
회수 완료
생존 02:14 / 처치 184
광고 전단 조각 +5
송출 기록: 잡음 속 주소 확보
도윤: 다음엔 도장 출력부터 손보자.
```

금지:

```text
인양만으로 보상 획득
trace_preservation +12
local_memory_profile = haunted
```

## 보급소 화면 최소 기능

필수 시설:

| 시설 | 기능 |
| --- | --- |
| 정산 카운터 | 런 결과 확인 |
| 정비대 | 윤서 기본 개조 |
| 이름 보관함 | 흔적 보관 |
| 출격 게시판 | 다음 지역/상태 확인 |
| 차징 조율대 | 반품 도장/회수선 계열 개조 |

0.2에서는 시설을 깊은 UI로 만들지 않아도 된다. 단, 보급소가 그냥 메뉴 목록처럼 보이면 실패다.

시각 변화 최소:

- 첫 개조 후 정비대 불빛.
- 흔적 보관 후 보관함 라벨.
- 송출 기록 2개 후 게시판 선.
- 보스 성과 후 코어 파편 유리병.

## 보스 처리

0.2 보스 선택:

```text
destroy_node
extract_memory
```

결절 파괴 필수 효과:

- `boss_outcome = "destroy_node"`
- `campaign_pressure` 감소.
- 게시판 `조용한 오픈하우스`.
- 미나 또는 복희 반응.
- 다음 런에 문패/안정화 느낌 증가.

기억 추출 필수 효과:

- `boss_outcome = "extract_memory"`
- `trace_preservation` 내부 상승.
- `sync_risk` 내부 소폭 상승.
- 게시판 `보정된 가족사진`.
- 복희 또는 세븐 반응.
- 다음 런에 가족사진/영수증/잔향 느낌 증가.

## 캐릭터 0.2

0.2 플레이어블:

```text
pc_yunseo_001
```

필수:

- 반품 스캐너 자동공격.
- 반품 도장 차징.
- 첫 회수 대사.
- 흔적 반응 최소 2개.
- 외곽 주택가 반복 방문 대사.

유저 선호 합격 기준:

```text
윤서가 "그냥 기본캐"로 보이지 않아야 한다.
```

0.2에서 팝시-제로, 나리, 고위험 역송출형은 실제 플레이어블로 구현하지 않는다. 단, 흔적과 NPC 반응으로 씨앗은 남긴다.

## 테스트 합격 기준

0.2 구현 후 반드시 확인:

- 첫 30초 안에 윤서 자동공격을 이해한다.
- 첫 차징이 시원하다.
- 첫 100초 회수가 억지 패배가 아니라 신호 과부하로 보인다.
- 첫 회수 후 바로 개조할 수 있다.
- 광고 전단 조각과 캠페인 코어 파편의 차이가 느껴진다.
- 송출 기록이 재화가 아니라 단서로 느껴진다.
- 외곽 주택가 재방문 시 게시판/적/대사 중 하나가 바뀐다.
- 보스 선택 후 다음 런 차이가 보인다.
- 내부 철학/점수/성향 UI가 노출되지 않는다.
- 윤서가 기본캐지만 계속 키우고 싶은 캐릭터로 보인다.

## 유저 선호 점수

0.2 구현 범위 예상:

| 기준 | 점수 | 이유 |
| --- | ---: | --- |
| 핵심 재미 전달 | 8 | 전투, 회수, 개조, 재방문이 모두 있음 |
| 구현 현실성 | 8 | 한 지역/한 캐릭터/한 보스로 제한 |
| 캐릭터 매력 | 8 | 윤서에 집중 가능 |
| 장기 확장성 | 9 | 지역/캐릭터/보급소 확장 구조가 열려 있음 |
| 리스크 | 7 | 손맛과 보급소 표현이 약하면 급락 |

종합:

```text
8.0 / 10
```

판정:

0.2 범위는 적절하다. 더 줄이면 세계의 깊이가 사라지고, 더 늘리면 구현이 터질 가능성이 높다. 유저가 좋아할 가능성을 높이는 핵심은 콘텐츠 수가 아니라 윤서 손맛, 첫 회수, 보스 선택 후 재방문 변화다.

## 금지 방향

- 0.2에서 플레이어블 캐릭터를 늘리지 않는다.
- 0.2에서 지역을 늘리지 않는다.
- 0.2에서 보스 선택지를 3개 이상 열지 않는다.
- 송출관을 실제 최종 보스로 만들지 않는다.
- 점수/성향/철학을 UI에 노출하지 않는다.
- 첫 회수 후 아무 개조도 못 하게 하지 않는다.
- 보스 조우만으로 코어 파편을 주지 않는다.
- 지역을 클리어 후 종료 상태로 만들지 않는다.
