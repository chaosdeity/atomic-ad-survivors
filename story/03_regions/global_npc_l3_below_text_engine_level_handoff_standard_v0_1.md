## 1. 핸드오프 목적

이 문서의 목적은 240개 적용 티켓을 실제 엔진 데이터로 즉시 변환하는 것이 아니라, 각 팀이 어떤 필드를 받아야 하는지 합의하는 것이다.

핸드오프 목적:

1. 엔진팀이 나중에 데이터 구조를 만들 때 필요한 최소 필드를 고정한다.
2. 레벨팀이 적용 위치를 오브젝트/표지/UI/오디오 트리거로 나눠 받을 수 있게 한다.
3. UI/오디오 팀이 표시 조건, 반복 조건, 자막/목록/호출번호 처리 기준을 이해하게 한다.
4. QA팀이 240개 티켓이 새 문구나 새 NPC로 번지지 않았는지 검증할 수 있게 한다.
5. 실제 JSON, CSV, Godot 리소스, GDScript 작성 전 단계로 멈춘다.

판정:

```text
이 문서는 엔진/레벨 전달 기준이며 실제 구현 파일이 아니다.
```

## 2. 데이터 역할

데이터 역할:

| 데이터 단위 | 역할 | 이번 문서 처리 | 금지 |
|---|---|---|---|
| `ticket_id` | 적용 후보 티켓 식별자 | 필수 필드로 고정 | 새 ticket_id 추가 |
| `sample_id` | 1차 샘플 문구 식별자 | 원본 240개와 1:1 유지 | 새 sample_id 작성 |
| `source_slot_id` | L3 이하 은행 출처 | 샘플 원문에서 참조 | 1,856개 전체 확장 |
| `text` | 원본 샘플 문구 | 원문 참조 필드로만 전달 | 새 문구 작성 |
| `placement_hint` | 배치 후보 위치 | 팀별 후보로만 전달 | 실제 레벨 배치 확정 |
| `state_key`/`flag` | 표시 조건 후보 | 문서용 연결 기준으로만 전달 | 엔진 플래그 즉시 생성 |

역할 판정:

```text
이 데이터는 source of truth가 아니라 handoff contract다.
엔진 데이터 패키지, CSV, Godot 리소스, GDScript로 변환하려면 별도 승인 문서가 필요하다.
```

1. 배치 후보는 실제 레벨 좌표가 아니다.

3. 티켓을 NPC 스폰, 상점, 강화, 튜토리얼, 퀘스트 제공자로 변환하지 않는다.
