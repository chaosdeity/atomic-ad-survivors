# R01 NPC 이름 레지스트리 v0.1

## 문서 상태

```text
상태:
스토리팀/시나리오팀 검토용 v0.1

용도:
R01 핵심 NPC 12명의 코드명, 역할명, 내부 본명/원본 이름 후보, 유저 노출 시점을 분리해 관리한다.

판정:
필요

주의:
이 문서는 코드명을 없애기 위한 문서가 아니다.
R01 초반 유저 노출은 여전히 OPEN-HOST, ROOM-12, PORCH-LINE 같은 역할명/코드명을 우선한다.
다만 이름/문패/주소/가족 등록이 핵심인 지역이므로, 내부 본명 후보는 중복 없이 관리한다.
```

---

## 0. 결론

NPC 이름은 지정해야 한다.

단, 모든 이름을 첫 등장부터 유저에게 보여주면 안 된다.

R01에서는 이름이 단순한 호칭이 아니라 등록, 문패, 주소, 가족 칸, 상담표와 연결된다.
그래서 이름은 다음 세 층으로 분리한다.

```text
1. 코드명:
개발/로스터/초기 유저 인식용.
예: ROOM-12, PORCH-LINE, FRAME-LEFT

2. 역할명:
유저가 처음 이해하는 기능적 이름.
예: 12번 방의 가족대표, 현관 줄 관리자, 가족사진 속 남은 사람

3. 내부 본명/원본 이름:
스토리팀이 중복 없이 관리하는 사람 이름.
대부분은 신뢰도, 흔적 보존, 복희 이름 보관, 보스 후 기억 추출 이후에 해금한다.
```

---

## 1. 중복 기준

전세계적으로 같은 이름이 있을 수는 있다.

하지만 R01의 실제 플레이 인구와 핵심 NPC 수는 작다.
따라서 유저가 직접 기억해야 하는 핵심 NPC 사이에서는 이름이 겹치면 손해가 크다.

권장 기준:

```text
1. R01 핵심 NPC 12명 사이에서는 같은 본명 금지.
2. 첫 시즌 주요 인물과 같은 이름 금지.
3. 같은 성씨는 가능하지만, 같은 이름/비슷한 호명감은 피한다.
4. 윤서, 미나, 복희, 도윤, 리오, PATCH, 세븐, 팝시와 혼동되는 이름은 피한다.
5. MAIL-LOOP, VAC-0처럼 사람이 아닌 NPC는 본명을 억지로 붙이지 않는다.
6. FRAME-LEFT처럼 프로필 잔향형 NPC는 완전 본명보다 "원본 이름 조각"으로 관리한다.
```

## 3. 유저 노출 원칙

R01 초반 유저 노출은 코드명/역할명이 먼저다.

이유:

```text
R01은 사람을 이름보다 역할로 읽는 지역이다.
유저도 처음에는 이들을 사람 이름보다 역할로 보게 된다.
그 불편함이 지역 감정이다.
```

하지만 끝까지 코드명만 남기면 안 된다.

이유:

```text
윤서와 복희의 축은 잘못 붙은 이름을 떼고, 이름을 보관하는 데 있다.
사람이 역할에서 조금씩 빠져나오려면,
언젠가는 이름 조각이 필요하다.
```

권장 노출 단계:

```text
초기:
코드명/역할명만 노출.

중반:
문패 조각, 상담표 조각, 순번표, 사진 뒤쪽 손글씨에서 이름 일부 노출.

후반:
복희 이름 보관함 또는 윤서 보류 선택 이후 본명/원본 이름 후보 해금.

주의:
본명을 해금해도 UI 주표기는 당분간 코드명+역할명을 유지할 수 있다.
예: ROOM-12 / 강연우
```

## 4. 이름으로 하면 안 되는 것

금지:

```text
코드명을 모두 한 번에 본명으로 교체.
유저 첫 만남부터 본명을 전부 공개.
이름을 알면 곧 해방된다는 식으로 처리.
이름 해금이 단순 수집 보상이 되는 구조.
로봇/잔향형 NPC에게 억지 사람 이름 부여.
비슷한 이름 반복으로 유저 기억을 흐리는 것.
```

특히 FRAME-LEFT는 주의한다.

```text
FRAME-LEFT의 이름은 "사진 속 사람의 정답"이 아니다.
사진 밖에 맡길 수 있는 이름 조각이다.
```

## 5. 다음 작업

이 문서는 v0.1 후보이므로, 다음 작업에서 아래를 확인한다.

```text
1. 각 이름이 기존 주요 캐릭터 이름과 충돌하지 않는가?
2. 같은 발음, 비슷한 호명감, 비슷한 감정 위치가 겹치지 않는가?
3. 각 이름이 해당 NPC의 해금 장면과 어울리는가?
4. 성별/나이 고정이 필요 이상으로 강해지지 않는가?
5. 본명 해금이 역할 해방으로 보이되, 즉시 구원으로 오해되지 않는가?
```
