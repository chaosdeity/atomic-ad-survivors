## 0. 결론

기존 1~13번 프로필과 결재 문서는 삭제하지 않는다. 다만 해당 숫자는 대부분 `프로필 산출 번호` 또는 `legacy_code`로 남긴다.

앞으로 플레이어블 제작팀은 아래 네 가지를 분리한다.

| 구분 | 의미 | 운용 |
|---|---|---|
| 프로필 산출 번호 | 문서가 작성된 순서 | 기존 파일명과 참조 보존용 |
| legacy_code | 이미 문서와 코드명에 붙은 숫자 | 즉시 변경하지 않음 |
| unlock_order | 캠페인 오픈 기준 해금 순서 | 별도 컬럼으로 관리 |
| user_display_order | 유저 UI 노출 순서 | 엔진/UX 단계에서 별도 판정 |

따라서 `GATE-09`가 새 해금 4번이 되어도 바로 `GATE-04`로 바꾸지 않는다. `GUARD-13`이 후기 E12 후보로 밀려도 기존 `guard_13_profile_v1_0.md` 파일명은 보존한다.

## 4. 마이그레이션 규칙

1. 이미 생성된 파일명은 이번 단계에서 바꾸지 않는다.
2. 새 해금 순번은 로스터와 레지스트리에 별도 컬럼으로 추가한다.
3. 코드명 숫자와 새 해금 순번이 다를 경우 `legacy_code`와 `unlock_order`를 나눠 쓴다.
4. `GATE-09`는 새 해금 4번 후보가 되어도 즉시 `GATE-04`로 바꾸지 않는다.
5. `GUARD-13`은 E12_C02 후보로 유지하되, 전태주는 최종명이 아니라 legacy profile name으로 둔다.
6. 14~19 한국어권 임시명은 최종명처럼 쓰지 않는다.
7. 한국인 신규 플레이어블 상세 결재는 별도 승인 전까지 금지한다.
8. 20번 OBS-20은 메타 시즌 관측자가 아니라 E03_C01 스마트단지 관제/입주 절차 후보로 둔다.

## 5. HP 0 판정 원칙

HP 0은 전투 사망이 아니라 캠페인의 제도적 판정이어야 한다.

| 계열 | HP 0 판정 방향 |
|---|---|
| E01 내부 후보 | 회수 실패, 퇴원 취소, 반송 처리, 충전 차단, 수신 공백처럼 각 캠페인 절차로 처리 |
| GATE/BROAD/CLAUSE/LEDGER | 개찰 미통과, 청취 확인 고정, 자동결제 승인, 미정산 장부 확정으로 처리 |
| RIO/GUARD | 객실 초기화/정품 복구, 보호자 서명 확정/치료 접근권 담보 고정으로 처리 |
| 14~20 교정 후보 | 해당 캠페인이 열린 뒤 로컬 절차를 먼저 확정하고 HP 0을 만든다 |
