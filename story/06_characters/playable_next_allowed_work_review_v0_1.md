## 0. 결론

이번 재검수는 후보를 더 만드는 작업이 아니다.
이미 만들어진 문서와 후보를 아래처럼 다시 읽는다.

1. 윤서는 기본 주인공으로 유지한다.
2. PATCH는 출시 초기 추가 1명 후보 가능성만 검토한다. `2번 고정`을 최종 해금 순번처럼 읽지 않는다.
3. OPEN-HOST는 R01 NPC 선등장 후 전환 조건만 검토한다.
4. HOLD, RETURN, METER, QUIET는 E01 한국인 추가 후보 과밀 위험 때문에 즉시 해금하지 않는다.
5. GATE, BROAD, CLAUSE, LEDGER, GUARD, MODEL은 각 원점 캠페인 개방 후 재검토한다.
6. LOCKOUT은 기능 초안으로만 보존한다.
7. NAME, RATION, LINE, OBS, TICKET은 캠페인 개방 전 상세 결재를 금지한다.
8. 13~19 한국어 임시명은 legacy placeholder로만 보존한다.

## 3. 출시 초기 제한 재확인

출시 초기 플레이어블은 아래처럼 다시 묶는다.

```text
기본 주인공:
윤서

추가 가능성:
최대 1~2명.
단, 이번 문서는 그 1~2명을 확정하지 않는다.

가장 먼저 검토할 수 있는 범위:
PATCH 초반 합류 조건.
OPEN-HOST NPC 선등장 후 전환 조건.

보류:
HOLD-04, RETURN-05, METER-06, QUIET-07은 프로필 산출물은 보존하되 출시 초기 즉시 해금으로 읽지 않는다.
```

이 제한은 기존 프로필을 삭제한다는 뜻이 아니다.
프로필 산출 번호, legacy_code, unlock_order, user_display_order를 분리한다는 뜻이다.

## 4. 번호 분리 잠금

앞으로 플레이어블 문서는 아래 네 값을 분리한다.

| 구분 | 의미 | 이번 문서의 잠금 |
|---|---|---|
| profile_output_no | 문서가 작성된 순서 | 기존 파일명과 참조 보존용 |
| legacy_code | 이미 문서에 붙은 코드명 숫자 | 즉시 변경하지 않음 |
| unlock_order | 캠페인 개방 후 실제 해금 순서 | 별도 표에서만 판정 |
| user_display_order | UI/도감/선택 화면 노출 순서 | 엔진/UX 단계에서 별도 판정 |

특히 `GATE-09`, `BROAD-10`, `CLAUSE-11`, `LEDGER-12`, `GUARD-13`, `MODEL-15`, `LOCKOUT-17`의 숫자는 해금 순번이 아니다.

## 6. 다음에 실제로 가능한 작업

이번 시점에서 실제로 가능한 작업은 새 후보 제작이 아니다.
가능한 작업은 출시 초기 플레이어블의 추가 허용 범위를 검토하는 것이다.

허용되는 작업 후보:

| 작업 | 가능 여부 | 이유 |
|---|---|---|
| 출시 초기 플레이어블 +1 후보 제한 검토 | 가능 | 윤서 기본 주인공 이후 추가 후보 수를 먼저 잠가야 PATCH/OPEN-HOST/E01 후보 과밀을 막을 수 있다. |
| PATCH 초반 합류 조건 재검토 | 가능 | 단독으로는 가능하지만, 출시 초기 +1 제한 검토 안에 포함하는 편이 안전하다. |
| OPEN-HOST NPC 선등장 후 전환 조건 검토 | 가능 | 단독으로는 가능하지만, 출시 초기 +1 제한 검토 이후에 전환 후보로 다루는 편이 안전하다. |
| 플레이어블 해금 순번/UI 노출 순서 분리표 작성 | 가능 | 필요하지만, 먼저 출시 초기 허용 폭을 정해야 표가 흔들리지 않는다. |

따라서 다음 작업은 하나만 남긴다.
