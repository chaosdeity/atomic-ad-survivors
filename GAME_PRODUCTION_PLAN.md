# Atomic Ad Survivors 제작 총괄 계획

## 0. 문서 목적

이 문서는 `Atomic Ad Survivors`를 Codex와 함께 여러 작업 단위로 나누어 제작하기 위한 기준 문서다.

이 방은 총괄 방으로 사용한다. 전체 방향, 우선순위, 작업 분해, 결과 검수는 이 문서를 기준으로 한다. 다른 Codex 방에서 개별 작업을 맡길 때는 이 문서와 해당 작업 지시서를 함께 전달한다.

## 1. 현재 프로젝트 요약

- 엔진: Godot 4.x, 현재 `project.godot` 기준 Godot 4.2
- 장르: 2D 탑다운 생존 로그라이크, `Vampire Survivors` / `Brotato` 계열
- 현재 메인 씬: `res://scenes/main.tscn`
- 현재 핵심 스크립트: `res://scripts/main.gd`
- 현재 톤: B - 밝은 디스토피아 원자력 교외
- 현재 아트 방식: 코드 드로잉 기반 임시 그래픽, 추후 스프라이트 교체 예정

한 줄 정의:

> 밝고 귀여운 1950s 광고 도시가 방사능과 기업 마스코트로 붕괴된 세계에서, 자동 전투의 편안함 위에 5초마다 찾아오는 짧은 차징 조준 순간으로 숙련도와 폭발적 보상을 주는 탑다운 생존 게임.

## 2. 확정된 핵심 방향

### 2.1 아트 톤

확정 톤은 `B - bright dystopian atomic`이다.

핵심 키워드:

- 1950s 광고지
- 파스텔 교외
- 활짝 웃는 기업 마스코트
- 귀엽지만 갈라진 표면
- 방사능 그린 슬라임
- 종이 카탈로그 UI
- 형광 그린/핑크/옐로우 이펙트

주요 레퍼런스 위치:

- `assets/reference_bright_dystopian/`
- `generated_assets/02_bright_dystopian_atomic/`

### 2.2 입력 모델

현재 주력 모델은 `M5_윈도우조준`이다.

규칙:

- 평소에는 자동 조준 + 자동 공격
- 플레이어는 이동 회피에 집중
- 5초마다 1초짜리 차징 윈도우 발생
- 윈도우 중 탭/클릭/스페이스 입력 시 광역 차징 공격
- 윈도우 중 마우스/터치 방향을 주면 더 강한 방향 집중 폭발

이 모델을 우선 검증하는 이유:

- 시뮬레이션상 스킬 천장이 가장 높음
- 입력 부담은 M2 차징 모델에 가깝게 낮음
- 캐주얼은 방향 조준을 무시해도 플레이 가능
- 미드코어/코어는 짧은 조준 순간으로 실력 표현 가능

## 3. 1차 목표 빌드

목표 이름: `Vertical Slice 0.1`

목표:

5분 동안 반복 플레이 가능한 단일 아레나 빌드를 만든다. 최종 에셋이 없어도 된다. 대신 코어 루프, 차징 감각, 레벨업 선택, 웨이브 압박, 결과 화면이 모두 작동해야 한다.

완료 조건:

- 시작 후 바로 플레이 가능
- 이동, 자동 사격, M5 차징 윈도우가 명확히 작동
- 5분 생존 목표 또는 사망까지 플레이 가능
- 레벨업 시 3개 카드 중 하나를 선택
- 카드 선택이 실제 능력치에 반영
- 적 웨이브가 시간에 따라 점진적으로 강해짐
- 게임오버/재시작/결과 화면 존재
- 성능이 적 150마리, 파티클 다수 상황에서도 크게 무너지지 않음

## 4. 코어 루프

1. 플레이어는 아레나에서 이동하며 적 군집을 피한다.
2. 기본 무기는 가장 가까운 적을 자동으로 공격한다.
3. 적 처치로 XP를 얻는다.
4. 5초마다 차징 윈도우가 열린다.
5. 플레이어는 윈도우에 맞춰 탭/클릭/스페이스로 차징 공격을 발동한다.
6. 조준 방향을 주면 더 강한 집중 폭발이 나간다.
7. XP가 일정량 쌓이면 레벨업 카드 3택이 뜬다.
8. 선택한 카드가 생존, 자동 공격, 차징 공격, 이동, 회복 중 하나를 강화한다.
9. 더 오래 버티며 더 많은 적을 처치한다.
10. 사망 또는 5분 생존 후 결과를 확인하고 다시 시작한다.

## 5. 현재 구현 상태

이미 구현됨:

- Godot 프로젝트 설정
- 단일 메인 씬
- 플레이어 이동
- 자동 사격
- 적 스폰과 추격
- 엘리트 적
- HP, 시간, 레벨, 처치 수, 적 수 HUD
- 5초 주기 1초 차징 윈도우
- 클릭/스페이스/터치 차징 발동
- 방향 조준 보너스
- 파티클, 화면 흔들림, 플래시
- 임시 레벨업 카드 패널
- 코드 생성 사운드
- 게임오버와 재시작

부족함:

- 모든 로직이 `scripts/main.gd` 하나에 몰려 있음
- 레벨업 카드가 실제 3택 구조가 아님
- 능력치/무기/웨이브 데이터가 하드코딩됨
- 적 종류가 단순함
- 보스가 없음
- 결과 화면이 약함
- 에셋이 실제 스프라이트로 연결되지 않음
- 사운드/음악 시스템이 임시 단계
- 모바일 터치 UI가 아직 검증되지 않음
- 성능 최적화 여지가 큼

## 6. 제작 우선순위

### P0 - 기반 정리

목표: 여러 Codex 방이 충돌 없이 작업할 수 있도록 구조를 나눈다.

작업:

- `main.gd`를 기능별 스크립트로 분리
- 게임 상태, 플레이어, 적, 투사체/이펙트, HUD, 레벨업, 오디오를 분리
- 공용 밸런스 값을 데이터 또는 상수 모듈로 이동
- 현재 동작을 유지한 상태로 리팩터링

완료 기준:

- 기존 플레이가 깨지지 않음
- `main.gd`가 총괄 역할만 담당
- 각 기능 파일의 책임이 명확함

### P1 - 재미 검증

목표: M5 차징 윈도우가 실제로 재미있는지 검증 가능한 빌드를 만든다.

작업:

- 차징 윈도우 진입 신호 강화
- 차징 성공/실패 피드백 강화
- 방향 조준 시 시각적 콘/폭발 개선
- 차징 윈도우 타이밍, 데미지, 타겟 수 조정 가능화
- 플레이테스트용 로그 또는 결과 수치 추가

완료 기준:

- 윈도우가 열렸다는 사실을 즉시 알 수 있음
- 발동 순간이 충분히 시원함
- 방향 조준이 왜 강한지 눈으로 이해됨

### P2 - 레벨업 카드와 빌드

목표: 반복 플레이의 선택 구조를 만든다.

초기 카드 후보:

- 자동 사격 DPS 증가
- 자동 사격 사거리 증가
- 차징 데미지 증가
- 차징 타겟 수 증가
- 차징 쿨다운 감소
- 이동 속도 증가
- 최대 HP 증가
- 즉시 회복
- XP 획득량 증가

완료 기준:

- 레벨업마다 3개 카드가 뜸
- 카드마다 이름, 설명, 효과가 있음
- 선택한 효과가 실제 게임에 반영됨
- 같은 카드를 여러 번 선택했을 때 누적 규칙이 명확함

### P3 - 웨이브, 적, 보스

목표: 5분 생존 흐름을 만든다.

작업:

- 시간대별 웨이브 테이블
- 잡몹/엘리트/특수 적 구분
- 3분 또는 5분 보스 등장
- 적별 HP, 속도, 접촉 피해, XP 보상 분리
- 보스 처치 또는 생존 결과 처리

완료 기준:

- 초반, 중반, 후반의 압박 차이가 있음
- 강한 빌드와 약한 빌드 차이가 체감됨
- 보스가 단순 큰 적이 아니라 패턴/존재감이 있음

### P4 - 에셋 파이프라인

목표: 코드 드로잉을 실제 에셋 기반으로 바꾼다.

작업:

- 필요한 스프라이트 목록 확정
- 이미지 생성/수정 프롬프트 작성
- 파일명 규칙과 폴더 구조 정리
- 플레이어, 잡몹, 엘리트, 보스, 타일, UI 카드, 아이콘 순서로 적용
- Godot import 설정 통일

권장 폴더:

```text
assets/
  characters/
    player/
    enemies/
      tier1/
      tier2/
      bosses/
  tilesets/
  ui/
  vfx/
  audio/
```

### P5 - 사운드와 음악

목표: 차징과 생존 압박의 감정 곡선을 만든다.

작업:

- 기본 BGM 루프
- 위기 레이어 또는 보스 레이어
- 차징 준비음
- 차징 발동음
- 적 처치음
- 피격음
- 레벨업음
- UI 선택음
- Godot 오디오 버스 구성

완료 기준:

- 차징 윈도우가 소리만 들어도 감지됨
- 발동음이 시원함
- 음악이 반복되어도 피로하지 않음
- 효과음 볼륨 밸런스가 무너지지 않음

### P6 - 최적화와 배포

목표: 오래 플레이해도 렉이 심하지 않은 빌드를 만든다.

작업:

- 적과 파티클 업데이트 비용 측정
- Dictionary 기반 핫루프 최적화 검토
- 파티클 수 제한
- 드로우 방식 개선
- Godot Web export 테스트
- 브라우저 실행 성능 확인

완료 기준:

- 데스크톱에서 5분 플레이가 안정적
- 적 150마리 상황에서 입력 지연이 크지 않음
- 웹 빌드에서 로딩과 실행이 가능

## 7. 권장 작업 분할 방식

이 방:

- 전체 설계
- 문서 관리
- 작업 지시서 작성
- 병합 전 검토
- 우선순위 조정

개별 Codex 방:

- 각자 한 작업만 담당
- 가능한 한 서로 다른 파일 범위를 담당
- 완료 후 변경 파일과 테스트 결과를 보고

중요 규칙:

- 여러 방이 동시에 같은 파일을 크게 수정하지 않는다.
- 특히 `scripts/main.gd`는 먼저 분리한 뒤 병렬 작업한다.
- 새 방에는 반드시 목표, 참고 파일, 수정 가능 파일, 완료 조건을 전달한다.
- 새 방 결과를 합치기 전 이 방에서 검토한다.

## 9. 첫 5개 권장 작업

1. `main.gd` 구조 분리
2. 레벨업 카드 3택 시스템 구현
3. M5 차징 윈도우 피드백 강화
4. 5분 웨이브 테이블과 결과 화면 추가
5. 에셋 목록과 생성 프롬프트 작성

권장 첫 작업은 `main.gd` 구조 분리다. 이 작업을 먼저 해야 이후 전투, UI, 에셋, 사운드를 다른 방에 나눠 맡기기 쉽다.

## 10. 현재 결정 대기 사항

다음 항목은 아직 확정하지 않아도 되지만, 제작 중 선택해야 한다.

- 최종 플랫폼: PC 우선인지, 모바일 우선인지, 웹 우선인지
- 1판 목표 길이: 5분, 10분, 15분 중 무엇인지
- 보스 등장 시점: 3분, 5분, 10분 중 무엇인지
- 메타 진행 여부: 판 밖 영구 성장 시스템을 넣을지
- 스토리 톤: 블랙코미디, 슬픈 광고풍, 순수 액션 중 어디에 무게를 둘지
- 에셋 생성 방식: AI 생성 후 보정, 직접 픽셀아트, 외부 툴 혼합 중 무엇인지

현재 추천:

- 1차는 PC + 웹 우선
- 1판 5분
- 3분 중간 엘리트, 5분 보스
- 메타 진행은 0.1 이후
- 스토리 톤은 밝은 광고풍 블랙코미디
- 에셋은 AI 생성 레퍼런스 + Godot 적용 가능한 픽셀 에셋 정리 방식
