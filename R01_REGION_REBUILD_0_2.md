# R01 Region Rebuild 0.2

## 0. 문서 목적

이 문서는 `Atomic Ad Survivors 0.2`의 R01 맵을 카드 엔진과 광고 감사 문턱에 맞게 다시 잡기 위한 기준서다.

핵심 결론은 명확하다.

> R01은 처음 제대로 테스트할 때부터 넓어야 한다.

작은 맵에서 먼저 테스트하고 나중에 넓히는 방식은 이 게임에는 맞지 않는다.

이유:

- 이 게임의 재미는 단순 전투장이 아니라 `위험 구역에 들어갈지 말지`의 판단에서 나온다.
- 카드 엔진은 `다음 감사 기준을 넘기기 위해 어디로 들어갈 것인가`와 연결되어야 한다.
- 맵이 작으면 위험 구역, 귀환 판단, 지역 기억, 광고 압박이 모두 한 덩어리로 뭉개진다.
- 작은 맵에서는 적 압박을 올려도 긴장이 아니라 혼잡으로 보일 가능성이 높다.

따라서 0.2 R01은 `좁은 아레나`가 아니라 `넓은 출격형 지역`으로 테스트해야 한다.

## 1. 현재 R01 판단

### 1.1 현재 블록아웃 규모

현재 `scripts/r01_layout_blockout.gd` 기준 R01 블록아웃은 다음 규모다.

```text
world_bounds:
  position: (-720, -405)
  size: 1920 x 810

viewport:
  480 x 270

camera_screens:
  4 x 3 = 12 screens
```

문서상으로는 12스크린이므로 작아 보이지 않을 수 있다.

하지만 실제 체감 이동 거리는 짧다.

현재 anchor 거리:

```text
silence_edge_start -> subdivision_loop_center: 514.97
subdivision_loop_center -> model_house_node_anchor: 670.06
subdivision_loop_center -> drain_pocket_anchor: 437.94
subdivision_loop_center -> fake_return_route_anchor: 458.60
```

현재 플레이어 속도:

```text
PLAYER_SPEED = 118.0
```

대략 이동 시간:

```text
침묵 가장자리 -> 분양 주택 루프: 약 4.4초
분양 주택 루프 -> 모델하우스 결절: 약 5.7초
분양 주택 루프 -> 배수로/가짜 귀환로: 약 3.7~3.9초
```

결론:

> 현재 R01은 구역 이름은 있지만, 지역 여행감은 아직 부족하다.

### 1.2 현재 적 압박의 한계

현재 기준:

```text
PLAYER_SPEED = 118.0
ENEMY_SPEED = 36.0
fast enemy role speed = 1.55
finale speed_mult = 1.40
```

빠른 일반 적의 후반 속도 예:

```text
36.0 * 1.55 * 1.40 = 78.12
```

플레이어 118보다 여전히 느리다.

물론 `charger` dash 같은 순간 압박은 있지만, 기본 구조는 여전히 플레이어가 거리를 벌리기 쉽다.

결론:

> 현재 구조에서는 맵이 작고, 적은 대체로 느리며, 압박은 공간 설계보다 스폰 밀도에 의존한다.

이 상태로는 `광고 감사 기준을 못 넘기면 공간이 닫힌다`는 카드 엔진의 핵심을 검증하기 어렵다.

## 2. 새 R01 규모 결정

### 2.1 첫 실전 테스트부터 넓게 간다

0.2의 첫 진짜 R01 플레이테스트는 아래 규모를 기준으로 한다.

권장:

```text
world_bounds:
  size: 3840 x 2160

viewport:
  480 x 270

camera_screens:
  8 x 8 = 64 screens
```

최소 허용:

```text
world_bounds:
  size: 3360 x 1890

viewport:
  480 x 270

camera_screens:
  7 x 7 = 49 screens
```

금지:

```text
48 camera screens 미만으로 R01 재미 검증을 하지 않는다.
```

이 기준은 최종 출시 맵 크기 확정이 아니다.

하지만 첫 재미 검증용 R01은 최소한 이 정도는 되어야 한다.

### 2.2 왜 64스크린인가

R01은 다음을 동시에 검증해야 한다.

1. 전투 손맛
2. 카드 선택
3. 위험 구역 진입
4. 감사 기준 압박
5. 귀환 판단
6. 태그 후보
7. 지역 상태 변화
8. 반복 출격의 차이

12스크린은 이 모든 것을 담기에는 너무 압축된다.

64스크린이면 다음이 가능하다.

- 안전 구역과 위험 구역을 실제 거리로 분리할 수 있다.
- 오픈하우스 거리를 "저 안쪽"으로 느끼게 할 수 있다.
- 모델하우스 접근로를 첫 5분의 최종 암시로 둘 수 있다.
- 배수로/가짜 귀환로 같은 보조 루트가 장식이 아니라 선택지가 된다.
- 적 압박이 단순 밀도가 아니라 동선 차단으로 작동한다.

### 2.3 목표 이동 시간

R01은 길기만 하면 안 된다.

이동 시간이 전투 판단과 섞여야 한다.

목표 이동 시간:

| 이동 | 목표 시간 |
| --- | ---: |
| 침묵 가장자리 -> 반복 주택가 | 10~16초 |
| 반복 주택가 -> 오픈하우스 거리 | 12~20초 |
| 오픈하우스 거리 -> 모델하우스 접근로 | 15~25초 |
| 반복 주택가 -> 배수로 보조 루트 | 8~14초 |
| 반복 주택가 -> 가짜 귀환로 | 10~16초 |
| 시작점 -> 모델하우스 접근로 직행 | 45~70초 |

중요:

> 플레이어가 달리기만 해서 모든 구역을 10초 안에 확인하면 안 된다.

## 3. R01 구조 결정: 2 + 3 + 보조 1

R01은 `2 + 3 + 보조 1` 구조로 간다.

이는 구역 개수가 아니라 설계 구조다.

```text
2 = 두 개의 주동선
3 = 세 개의 큰 지역 덩어리
보조 1 = 하나의 우회/위험 보조 네트워크
```

### 3.1 두 개의 주동선

#### A. 생존/정산 주동선

```text
침묵 가장자리
-> 반복 주택가
-> 배수로/귀환 판단
```

역할:

- 초반 전투 안정
- 식량태그/충전태그 후보
- 낮은 압력 정산
- 귀환 판단

플레이어 감정:

> 오늘 보급소가 버틸 만큼은 챙길 수 있겠다.

#### B. 신호/위험 주동선

```text
반복 주택가
-> 오픈하우스 거리
-> 모델하우스 접근로
```

역할:

- 수신태그 후보
- 감사 기준 처리량 증폭
- 캠페인 압력 증가
- 모델하우스 결절 암시

플레이어 감정:

> 위험하지만, 저 안쪽에 다음 판을 바꿀 게 있다.

### 3.2 세 개의 큰 지역 덩어리

#### 1. 침묵 가장자리

보급소와 아직 연결된 바깥쪽 지역.

역할:

- 첫 전투
- 첫 차징 손맛
- 낮은 광고 농도
- 귀환 기준점

금지:

- 너무 오래 머물러도 보상이 충분하면 안 된다.
- 안전 지대가 영구 파밍 구역이 되면 안 된다.

#### 2. 반복 주택가

R01의 가장 넓은 기본 전투 지역.

역할:

- 첫 카드 선택 이후 빌드 검증
- 반품 표식/전단/로봇 처리 실험
- 식량태그/충전태그 후보
- 오픈하우스 방향 노출

필수 감각:

> 같은 집이 반복되지만, 같은 길은 아니다.

#### 3. 오픈하우스 거리 + 모델하우스 접근로

0.2 R01의 핵심 위험 구역.

역할:

- 수신태그 후보
- 처리량 승수
- 캠페인 압력 상승
- 스마일 홈 시어머니 암시
- 보스/결절 후속 연결

필수 감각:

> 여기서 버티면 다음 감사 기준을 넘길 힘이 생기지만, 돌아가기 어려워진다.

### 3.3 보조 1: 우회/위험 보조 네트워크

보조 네트워크는 단일 구역명이 아니라, 아래 기능을 묶는 시스템이다.

```text
배수로 침묵 주머니
가짜 귀환로
문패/우편함 샛길
끊긴 광고 산책로
```

역할:

- 위험하지만 짧은 우회
- 충전태그/흔적/정산 사유
- 캠페인이 플레이어 동선을 학습했다는 감각
- 귀환과 혼동되는 가짜 길

주의:

- 실제 귀환 UI와 절대 헷갈리면 안 된다.
- 완전한 안전 루트가 되면 안 된다.
- 입구와 출구가 모두 막히는 불공정 함정이 되면 안 된다.

## 5. 광고 감사와 맵 연결

R01 맵은 `CARD_AND_RULE_ENGINE_0_2.md`의 광고 감사 기준과 직접 연결된다.

### 5.1 구역별 감사 역할

| 구역 | 감사 역할 | 기준 미달 시 |
| --- | --- | --- |
| 침묵 가장자리 | 정상 고객 예비 감사 | 검문 로봇 소폭 증가 |
| 반복 주택가 | 전단 확산/정산 예비 감사 | 적 밀도 증가, 빠른 전단 적 증가 |
| 오픈하우스 거리 | 오픈하우스 방문/가족 적합성 감사 | 빠른 검수 로봇, 오픈하우스 압력 증가 |
| 모델하우스 접근로 | 모델하우스 신호/결절 예비 감사 | 회수선 안정도 감소, 포위형 웨이브 |
| 보조 네트워크 | 회수선/흔적 감사 | 귀환 혼선, 후보 보류 위험 |

### 5.2 위험 구역은 성장 압축 구역이다

오픈하우스 거리는 단순 고난도 구역이 아니다.

오픈하우스 거리는 다음 감사 기준을 넘기기 위한 성장 압축 구역이다.

```text
오픈하우스 거리 진입
-> 캠페인 압력 증가
-> 적 밀도/속도 증가
-> 수신태그 후보와 처리량 승수 증가
-> 다음 감사 기준 돌파 가능성 증가
```

따라서 오픈하우스를 피하면 당장은 안전하지만, 후반 감사 기준에서 밀려야 한다.

### 5.3 기준 미달은 공간을 닫아야 한다

감사 기준 미달은 단순 정산 패널의 패널티가 아니다.

즉시 공간 압박으로 이어져야 한다.

예:

| 미달 상태 | 맵 반응 |
| --- | --- |
| 1회 미달 | 플레이어 후방 스폰 증가 |
| 2회 미달 | 오픈하우스 방향에 빠른 검수 로봇 배치 |
| 3회 미달 | 귀환 방향에 광고 산책로 혼선 발생 |
| 연속 미달 | 보조 네트워크 일부가 위험 루트로 변함 |

핵심:

> 도망은 시간을 벌 수 있지만, 기준 미달을 해결하지는 못한다.

## 6. 적 속도와 압박 기준

R01이 넓어지면 적 압박도 다시 잡아야 한다.

### 6.1 기본 원칙

모든 적이 항상 플레이어보다 빠르면 불공정하다.

하지만 모든 적이 플레이어보다 느리면 긴장이 없다.

따라서 적은 역할별로 속도 계층을 나눈다.

| 적 역할 | 목표 속도 |
| --- | --- |
| 기본 광고 적 | 플레이어 속도 45~65% |
| 빠른 전단/검문 적 | 플레이어 속도 75~92% |
| 감사 미달 추적 적 | 플레이어 속도 95~108%, 짧은 지속 |
| 돌진 적 | 예고 후 순간적으로 140~180% |
| 엘리트/보스 소환 | 느리지만 공간을 막거나 구역을 장악 |

### 6.2 압박은 속도만이 아니다

적 속도만 올리면 모바일에서 불쾌해진다.

압박은 아래 요소를 섞어 만든다.

- 빠른 적
- 후방 스폰
- 측면 포위
- passable hazard
- route decal 혼선
- 오픈하우스 체류 압력
- 회수선 안정도 감소

### 6.3 첫 5분 목표

첫 5분 안에 플레이어가 최소 한 번은 느껴야 한다.

```text
이 조합으로는 다음 기준을 못 넘길 수도 있다.
저 안쪽에 들어가면 위험하지만, 들어가야 힘을 얻는다.
도망만 치면 결국 밀린다.
```

## 7. 구역별 설계 상세

### 7.1 침묵 가장자리

역할:

- 시작 완충
- 첫 조작/차징
- 보급소와 연결된 안정감
- 첫 감사 기준

보상:

- 확정 식량태그 조건의 씨앗
- 낮은 위험 충전태그 후보 소량

압박:

- 낮음
- 오래 머물면 보상 효율 하락

카드 연결:

- 반품 라벨
- 야근 정산표
- 정상 고객 스티커

### 7.2 반복 주택가

역할:

- 메인 전투 루프
- 카드 조합 확인
- 식량태그/충전태그 후보 생성
- 오픈하우스 방향 제시

보상:

- 식량태그 후보
- 충전태그 후보
- 반품 표식/전단/로봇 명령 정산 사유

압박:

- 중간
- 감사 미달 시 적 밀도 증가

카드 연결:

## 8. 반복 출격 변화

R01은 한 번 쓰고 버리는 스테이지가 아니다.

같은 큰 맵 안에서 출격마다 강조 구역이 달라져야 한다.

| 상태 | 맵 변화 |
| --- | --- |
| 첫 방문 | 침묵 가장자리와 반복 주택가 중심 |
| 수신태그 확보 | 오픈하우스 방향 표식 증가 |
| 충전태그 배분 | 정비 관련 우회/로봇 명령 오류 이벤트 증가 |
| 식량태그 배분 | 인간 흔적/문패 이벤트 증가 |
| 캠페인 압력 상승 | 가짜 귀환로, 빠른 검수 로봇, 후방 스폰 증가 |
| 모델하우스 노출도 상승 | 접근로 선명화, 보스/결절 문구 증가 |

중요:

> 맵이 넓어야 반복 출격 변화가 실제 위치 변화로 느껴진다.

작은 맵에서는 변화가 전부 같은 화면의 장식 변경처럼 보인다.

## 12. 1차 구현 반영 기록

### 12.1 구현 반영 파일

- `scripts/r01_map_assembly.gd`

### 12.2 실제 적용 규모

문서의 권장 기준은 `3840 x 2160`, 64 camera screens였다.

실제 1차 구현은 이보다 더 넓게 잡았다.

```text
world_bounds:
  position: (-2640, -1485)
  size: 5280 x 2970

viewport:
  480 x 270

  11 x 11 = 121 screens
```

- 시작점에서 모델하우스 접근까지 최소 45초대의 지역 이동감을 만들기 위해서.
- 작은 맵에서 압박을 테스트하면 도망/혼잡/스폰 문제가 섞여 판단이 왜곡되기 때문에.
- 오픈하우스 거리와 보조 네트워크를 한 화면 장식이 아니라 실제 선택지로 만들기 위해서.

## 13. 2차 구현 반영 기록

### 13.1 world-space zone 판정

R01 진행 구역 판정은 이제 시간만 보지 않는다.

`R01LayoutBlockout.ENABLED` 상태에서는 플레이어 위치에서 가장 가까운 R01 anchor를 현재 구역으로 사용한다.

의미:

- 플레이어가 오픈하우스 거리로 직접 들어가야 오픈하우스 체류가 쌓인다.
- 모델하우스 접근로는 시간 경과가 아니라 실제 이동으로 감지된다.
- 넓은 맵의 보조 루트와 위험 구역이 정산/압박 시스템에 연결될 수 있다.

### 13.2 오픈하우스 체류 정산

오픈하우스 거리는 첫 구현 기준으로 다음 문턱을 가진다.

```text
25초 체류: 수신태그 후보 감지
50초 체류: 수신태그 승인 보정
75초 체류: 모델하우스 신호 선명화
```

현재 구현상 이 문턱은 `signal_clue_candidates`, 처리량 승수, 정산 문구에 연결된다.

```text
25초 체류: 처리량 x1.08
50초 체류: 처리량 x1.16
75초 체류: 처리량 x1.25
```

정산 예시:

```text
오픈하우스 처리량 보정: x1.16 (처치 90 -> 104 판정)
오픈하우스 체류 정산: 50초 - 수신태그 승인 보정이 붙었습니다.
신호 단서 판정: faint_signal, detected_signal
```

### 13.3 오픈하우스 압박 프로필

오픈하우스 거리에서는 일반 구역보다 압박이 커진다.

현재 구현 기준:

```text
spawn_pressure x1.18
speed_mult x1.06
elite_chance +0.04
fast/signal/speaker/charger 비중 증가
```

목표는 단순히 적을 많이 내는 것이 아니다.

오픈하우스 거리는 “수신태그 후보를 얻기 위해 들어가지만, 오래 머물수록 처리량 기준을 못 넘기면 밀리는 구역”이어야 한다.

### 13.4 이번 구현으로 닫힌 과제

닫힌 과제:

- 시간 기반 zone 판정을 world-space zone 판정으로 통합.
- 오픈하우스 체류를 수신태그 후보/정산 사유로 연결.
- 오픈하우스 거리의 적 압박 프로필 1차 적용.

다음 과제:

1. 카드/효과 카드가 오픈하우스 체류 보상을 증폭하도록 연결.
2. 광고 감사 기준과 처리량 미달 압박을 실제 전투 템포에 연결.
3. 후방 스폰/측면 포위 구조를 넓은 맵 기준으로 재조정.
4. 정산 UI에서 식량태그/충전태그/수신태그 후보를 기준명으로 표시.

### 13.5 검증 결과

Godot headless 실행:

```text
PASS
```

확인된 것:

- `5280 x 2970` R01 world bounds 유지.
- `camera_screens = 121.0` 유지.
- `open_house_route`, `model_house_node`, `fake_return_route` probe 통과.
- `30/100/300` density pathing preview 통과.
- 오픈하우스 zone/정산 연결 코드가 런타임 시작 단계에서 오류 없이 로드됨.

## 14. 3차 구현 반영 기록

### 14.1 보조 네트워크 지역 약관 1차 적용

R01의 보조 네트워크가 단순 이동 구역이 아니라 정산 규칙을 바꾸는 지역 약관으로 연결됐다.

적용 구역:

- `drain_pocket_anchor`
- `fake_return_route_anchor`

### 14.2 배수로 침묵 주머니

`drain_pocket_anchor`는 조용한 신호 우회 구역으로 잡는다.

현재 규칙:

- 20초 이상 체류 시 수신태그 후보 +1
- 수신태그 승인 가능성 소폭 상승
- 차징/장판/로봇 처리 계열 처리량 보정
- 감사 기준은 약간 낮아지지만, 압박이 사라지는 안전 구역은 아니다

정산 문구:

```text
지역 약관: 배수로 침묵 주머니 20초 - 수신태그 후보가 조용히 올라왔습니다.
```

### 14.3 끊긴 광고 산책로

`fake_return_route_anchor`는 귀환처럼 보이지만 실제 귀환은 아닌 가짜 안정 루트로 잡는다.

현재 규칙:

- 20초 이상 체류 시 식량태그 후보 +1
- 식량태그 승인 안정성 소폭 감소
- 감사 압력이 있으면 식량태그 오염 위험
- 차징/반품 표식 처리량은 오르지만 감사 기준도 더 높아진다

정산 문구:

```text
지역 약관: 끊긴 광고 산책로 20초 - 식량태그 후보와 오염 위험이 함께 생겼습니다.
```

### 14.4 의미

이번 구현으로 보조 네트워크의 방향이 분리됐다.

## 15. 수신태그 배분에 따른 지도 표시 1차 구현 기록

### 15.1 목적

R01은 넓어진 만큼, 플레이어가 “어디로 가야 하는지”를 최소한으로 읽을 수 있어야 한다.

다만 모든 길을 자동으로 안내하면 지역 탐험과 위험 구역 욕심이 약해진다.
따라서 수신태그를 출격 게시판에 배분했을 때만 지도 해석이 강해지는 구조로 둔다.

### 15.2 구현된 표시

수신태그 배분 후 R01 출격 중 다음 표시가 활성화된다.

- 플레이어 주변 방향 나침반
- 오픈하우스 수신태그 후보 핀
- 수신태그 2장 이상 배분 시 모델하우스 신호 핀
- 분양 주택 루프 -> 오픈하우스 거리 경로선
- 오픈하우스 거리 -> 모델하우스 접근로 경로선

### 15.3 단계별 의미

| 수신태그 배분 | R01 지도 감각 |
| --- | --- |
| 0 | 광고 구역을 직접 읽어야 함 |
| 1 | 오픈하우스 방향이 게시판에 붙잡힘 |
| 2 | 오픈하우스 이후 모델하우스 경로가 보이기 시작 |
| 3 이상 | 모델하우스 결절 접근 핀이 먼저 깜박임 |

### 15.4 검증 포인트

확인할 것:

1. 넓어진 맵에서 플레이어가 너무 빨리 길을 잃지 않는가?
2. 수신태그를 쓴 뒤 다음 출격 변화가 화면에서 느껴지는가?
3. 방향 표시가 정답 화살표가 아니라 위험 구역 유혹으로 읽히는가?

## 16. 지역 약관 예고 1차 구현 기록

### 16.1 목적

이번 구현은 플레이어가 구역에 들어간 뒤 정산에서만 규칙을 알게 되는 문제를 줄이기 위해, 지역 약관을 미리 예고하는 흐름을 추가했다.

### 16.2 구현된 예고

| 표시 위치 | 역할 |
| --- | --- |
| 보급소 출격 게시판 | 다음 출격의 대표 약관을 미리 보여줌 |
| 런 시작 배너 | 출격 직후 이번 런의 약관을 요약 |
| 구역 진입 배너 | 해당 구역 약관을 한 줄로 안내 |
| 정산 HUD | 현재 구역 약관을 짧은 태그로 표시 |

### 16.3 구역별 약관

| 구역 | 예고 문구 방향 |

| 배수로 침묵 주머니 | 20초 체류=수신태그 후보, 조용한 보정 |
| 끊긴 광고 산책로 | 20초 체류=식량태그 후보, 오염 위험 |
| 모델하우스 접근로 | 결절 신호 감사 강화 |

### 16.4 검증 포인트

1. 약관 예고가 구역의 성격을 즉시 구분하게 만드는가?
2. 플레이어가 위험 구역 진입을 “알고 선택했다”고 느끼는가?
3. 약관 문구가 너무 길어서 모바일 화면에서 잘리지 않는가?

## 17. 오염 결과의 지역 상태 반영 1차 구현 기록

### 17.1 목적

R01은 단순 반복 스테이지가 아니라 지역 RPG 공간이어야 한다.

따라서 오염된 태그 정산 결과는 정산 화면에서 끝나지 않고, 다음 런의 R01 상태로 남는다.

### 17.2 구현된 지역 변화

오염 결과는 다음 지역 상태를 바꾼다.

| 오염 종류 | 지역 변화 |
| --- | --- |
| 식량태그 오염 | 끊긴 광고 산책로의 감사/오염 위험 강조 |
| 충전태그 오염 | 분양 주택 루프의 로봇 검수 압력 증가 |
| 수신태그 오염 | 오픈하우스 거리와 모델하우스 접근로의 신호 감사 증가 |

### 17.3 전투 변화

오염 누적은 다음 전투 압력을 만든다.

- 전체 스폰 압력 소폭 증가
- 적 이동 속도 소폭 증가
- 엘리트 출현 판정 소폭 증가
- 식량태그 오염은 빠른 광고/오염 경로 압박을 늘림
- 충전태그 오염은 장갑/로봇 계열 압박을 늘림
- 수신태그 오염은 신호/스피커 계열 압박을 늘림

### 17.4 맵 표시

오염은 월드 위에 표식으로 표시된다.

| 표식 | 위치 |
| --- | --- |
| 오염 식량태그 | 끊긴 광고 산책로 |
| 오염 충전태그 | 분양 주택 루프 |
| 오염 수신태그 | 오픈하우스 거리 |
| 신호 감사 | 신호 오염 누적 또는 결절 접근 시 모델하우스 접근로 |

### 17.5 의미

이제 R01은 다음 구조를 가진다.

```text
플레이어의 정산 결과
-> 지역 상태 누적
-> 다음 런 노선 문구 변화
-> 다음 런 약관/스폰 압력 변화
-> 맵 위 흔적 표시
```

검증할 질문:

1. 오염 표식이 지역 변화를 충분히 보여주는가?
2. 오염 압력 증가가 너무 작아 무의미하거나 너무 커서 처벌처럼 보이지 않는가?
3. R01의 넓은 맵에서 오염 표식이 탐험 목표 또는 회피 목표로 읽히는가?
