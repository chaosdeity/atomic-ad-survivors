# RPG Growth Readiness Report

작성일: 2026-05-17
목적: RPG 성장/경제 구조를 넣기 전 현재 준비도와 막힐 지점을 판단한다.

## 1. 현재 준비도 요약

현재 게임은 RPG 성장 구조의 프로토타입 발판은 있다. 그러나 아직 “성과 기반 성장 경제”는 구현되어 있지 않다.

준비된 것:

- 세션 내 흔적/재화 컨테이너: `MetaProgression`.
- 보급소 화면과 강화 버튼.
- 런 중 카드 성장.
- 보스 회수/처치 보상.
- route/goal UI.
- 수치 시뮬레이터.

부족한 것:

- 런 종료 보상 공식.
- 빠른 죽음 파밍 방지.
- 신호 단서 3종의 실제 state.
- 보급소 업글 트리.
- 업글 비용 증가/해금 조건.
- 성과와 UI 문구를 연결하는 정산 리포트.

## 2. RPG 성장 구조 도입 전 선행 작업

### 선행 작업 1. 런 성과 정산 모델

필요 이유:

죽거나 회수됐다는 사실만으로 재화를 주면 빠른 죽음이 최적화된다. 보상은 생존 시간, 처치, 엘리트 처치, 신호 이벤트, 보스 HP 성과에 따라 계산되어야 한다.

필요 산출물:

- `RunResult` 데이터 구조.
- `RunReward` 데이터 구조.
- `evaluate_run_result()` 함수.
- 결과 화면에 표시할 보상 라인.

### 선행 작업 2. 신호 단서 진행도

현재 `신호 추적 1/3`은 UI 문구이고, 실제 단서 inventory가 아니다.

필요 산출물:

- `faint_broadcast_record`
- `warped_broadcast_map`
- `broadcast_coordinate_trace`
- 각 단서 최초 획득/중복 보상 규칙.
- 단서 3개 보유 시 보스 조우 가능.

### 선행 작업 3. 보급소 업글 트리 확장

현재 강화는 4개뿐이다.

필요 산출물:

- 최소 12개 업글.
- 비용 재화 종류.
- 최대 레벨.
- 해금 조건.
- 반복 비용 증가.
- 현재 `MetaProgression.UPGRADES` 구조 확장 또는 데이터 분리.

### 선행 작업 4. sortie_index 의존 약화

현재 보스 루트는 `sortie_index`, 시간, `boss_signal_unlocked`에 기대고 있다.

필요 방향:

- 출격 횟수는 표시/디버그 용도에 가깝게 낮춘다.
- 보스 해금은 단서와 해당 런 도달 조건 중심으로 바꾼다.
- 240초 도달은 유지하되 “단서 3개 + 240초 도달”이 되어야 한다.

### 선행 작업 5. 보급소 UI 확장 준비

성과 정산과 업글 트리를 넣으면 현재 보급소 패널은 부족하다.

필요 방향:

- 결과 정산 화면과 보급소 강화 화면을 분리.
- 보급소를 최소 2탭 또는 2상태로 나누기.
- 강화 목록은 4개 고정 버튼에서 스크롤/페이지/카테고리 형태로 확장.

## 3. 현재 성장 데이터와 한계

현재 `meta_progression.gd` 상태:

| 데이터 | 현재 역할 | 한계 |
| --- | --- | --- |
| `torn_ad_flyer` | 일반 보급소 강화 비용 | 획득 공식이 거의 없음 |
| `campaign_core_fragment` | 보스 대응 강화 비용 | 보스 전까지 의미 없음 |
| `boss_analysis_level` | 보스 회수 분석 단계 | 신호추적과 분리되어 있음 |
| `boss_clear_count` | 보스 처치 횟수 | 다음 지역/지역 변화와 연결 없음 |
| `awarded_flags` | 중복 지급 방지 | 이벤트가 늘면 문자열 플래그 난립 위험 |

## 4. 성장 경제가 막힐 지점

1. **보상 지급 지점이 흩어져 있다.**
   첫 회수, 보스 회수, 보스 승리에서 직접 지급한다. 일반 런 성과 정산은 없다.

2. **결과 화면이 보상 정산을 담기에 좁다.**
   생존 시간/처치/카드/흔적/단서/재화를 모두 보여주기 어렵다.

3. **업글 버튼이 4개 고정 구조다.**
   12개 이상 업글을 넣으려면 UI 구조 변경이 필요하다.

4. **카드 성장과 영구 성장이 같은 수치에 영향을 준다.**
   둘을 구분하는 표시와 밸런스 기준이 필요하다.

5. **웨이브가 시간표 중심이다.**
   성장한 플레이어가 깊이 들어가는 구조는 가능하지만, 신호 단서 획득과 웨이브 이벤트 연결이 아직 없다.

전체 판단:

```text
RPG 성장 구조 구현 가능 상태지만,

먼저 정산/단서/보급소 UI 경계를 잡아야 한다.
```

## 6. 권장 구현 순서

1. `run_result_evaluator.gd` 설계 및 추가.
2. 신호 단서 state를 `meta_progression.gd` 또는 `route_progression.gd`에 추가.
3. 결과 화면에 정산 라인 표시.
4. 보급소 UI를 업글 목록 확장 가능하게 개편.
5. 업글 12개를 데이터로 추가.
6. 보스 조우 조건을 단서 기반으로 전환.
7. `balance_sim.py`에 성과 보상/업글 비용 preview 추가.

## 7. PMO 판단 필요

- 첫 강제 회수 보상량을 얼마로 할지.
- 120/180/240초 단서가 “도달만”으로 충분한지, 특정 이벤트 성공 조건이 필요한지.
- 보스 조우까지 평균 몇 런을 목표로 할지.
- 0.2 버전에서 저장/불러오기를 계속 제외할지.
- 보급소 UI를 임시 코드 UI로 확장할지, scene 분리를 시작할지.
