# Ration Ticket Settlement 0.2

## 0. 문서 목적

이 문서는 `Atomic Ad Survivors 0.2`의 배급권과 정산 시스템을 정의한다.

목표는 단순 보상표를 만드는 것이 아니다. 목표는 플레이어가 다음을 이해하게 하는 것이다.

> 내가 어떤 구역에 들어갔고, 어떤 카드와 차징을 썼고, 어떤 위험을 감수했기 때문에 밥표/전원표/신호표 정산이 이렇게 나왔다.

이 문서는 `PROJECT_DIRECTION_LOCK_V1.md`와 `FIRST_5_MIN_FUN_AND_RPG_LOOP_0_2.md`를 기준으로 한다.

## 1. 핵심 원칙

### 1.1 배급권은 최상위 생존 자원이다

0.2에서 플레이어가 가장 먼저 이해해야 할 최상위 자원은 다음 3개다.

| 자원 | 의미 | 주요 역할 |
| --- | --- | --- |
| 밥표 | 인간 생존자 배급권 | 인간 안정도, 보급소 생존 |
| 전원표 | 해방 로봇 작동권 | 로봇 안정도, 정비/지원 기능 |
| 신호표 | 송출/결절 해석권 | 지역 해석, 모델하우스 접근, 다음 목표 |

이 자원은 골드가 아니다.

이 자원은 누가 하루 더 버틸지, 다음 런에 어떤 기능이 살아 있을지, 지역 접근이 얼마나 빨라질지를 바꾸는 생존권이다.

### 1.2 정산은 행동 기반이다

정산은 랜덤 보상표가 아니다.

정산은 런 중 행동이 만든 결과다.

```text
런 행동
-> 후보 생성
-> 캠페인 압력/카드/구역 조건 반영
-> 승인/보류/손실 판정
-> 보급소 배분
-> 다음 런 변화
```

### 1.3 생존권은 판매하지 않는다

밥표/전원표/신호표는 실제 과금 상품으로 직접 판매하지 않는다.

금지:

- 돈으로 밥표 구매
- 돈으로 전원표 구매
- 돈으로 신호표 구매
- 광고 시청으로 밥표 지급
- 광고 시청으로 전원표 지급
- 광고 시청으로 신호표 지급
- 광고 시청으로 정산 실패 무효
- 광고 시청으로 보급소 생존

## 2. 기존 재화와의 관계

기존 문서의 `광고 전단 조각`, `캠페인 코어 파편`, `송출 기록`, `흔적`은 폐기하지 않는다.

다만 플레이어가 첫 5분에 보는 최상위 보상은 배급권 3종으로 재정렬한다.

| 항목 | 새 위치 | 역할 |
| --- | --- | --- |
| 밥표 | 최상위 생존 자원 | 인간 생존/안정 |
| 전원표 | 최상위 생존 자원 | 로봇 작동/정비 |
| 신호표 | 최상위 진행 자원 | 지역 해석/결절 접근 |
| 광고 전단 조각 | 하위 성장 재료 | 장비/보급소/개조 재료 |
| 캠페인 코어 파편 | 결절/보스 하위 재료 | 상위 개조/보스 대응 |
| 송출 기록 | 스토리/해석 단서 | 지역 구조/상위 결절 접근 |
| 흔적 | RPG 감정/지역 반응 물건 | NPC 반응/지역 상태/기억 보존 |

정리:

```text
배급권 = 오늘 살아남는 이유
전단/코어 = 성장과 제작의 재료
송출 기록 = 이해와 접근의 단서
흔적 = 세계와 관계가 남는 물건
```

## 3. 배급권 3종 상세

### 3.1 밥표

정의:

> 인간 생존자 배급권.

플레이어 감각:

> 오늘 보급소의 인간 구역이 버틸 수 있는가?

획득 후보 조건:

- 일정 시간 이상 생존
- 낮은 체력 상태에서 버팀
- 인간 흔적 보존
- 반품 표식 적 처치
- 위험 구역에서 무리하지 않고 귀환
- 밥벌이 카드 `야근 정산표`, `저체력 근무수당`

사용처:

- 인간 구역 배분
- 시작 회복/지원 후보 증가
- 미나/복희 계열 반응
- 인간 안정도 변화

부족 시 영향:

- 보급소 회복/휴식 지원 약화
- 인간 구역 분위기 악화
- 일부 인간 NPC 불안 반응

### 3.2 전원표

정의:

> 해방 로봇 작동권.

플레이어 감각:

> 로봇이 꺼지지 않고 자기 이름으로 움직일 수 있는가?

획득 후보 조건:

- 로봇 적 처리
- 로봇 명령 오류 상태 처리
- 검문 로봇 우회/무력화
- 전원 관련 흔적 확보
- 밥벌이 카드 `배터리 절약 영수증`
- 도구 카드 `로봇 명령 뒤집기`

사용처:

- 정비대/충전 스탠드 배분
- 차징/도구 카드 출현 보정
- 팝시/세븐 계열 반응
- 로봇 안정도 변화

부족 시 영향:

- 정비대 기능 약화
- 로봇 지원 이벤트 감소
- 해방 로봇 동작 정지 이벤트 가능

### 3.3 신호표

정의:

> 송출/결절 해석권.

플레이어 감각:

> 다음 지역과 모델하우스 결절을 읽을 수 있는가?

획득 후보 조건:

- 오픈하우스 거리 체류
- 송출 잔류로 처치
- 전단 폭발/신호 이벤트 성공
- 위험 구역 깊이 진입
- 신호 관련 흔적 확보
- 밥벌이 카드 `오픈하우스 방문 확인`

사용처:

- 출격 게시판에 고정
- 오픈하우스 신호 위치 표시
- 모델하우스 노출도 상승
- 스마일 홈 시어머니 접근 암시
- 세븐 계열 해석 반응

부족 시 영향:

- 지역 해석 지연
- 보스 접근 지연
- 다음 런 목표 표시 약화

## 4. 확정 배급권과 후보 배급권

### 4.1 왜 두 층이 필요한가

배급권은 완전 랜덤이면 억울하고, 완전 고정이면 재미가 줄어든다.

따라서 0.2 정산은 두 층으로 둔다.

```text
확정 배급권
후보 배급권
```

### 4.2 확정 배급권

확정 배급권은 조건 달성 시 반드시 얻는다.

예:

- 첫 검증 런에서 90초 이상 생존 후 귀환: 밥표 1장 확정
- 특정 튜토리얼 목표 달성: 밥표 1장 확정
- 보급소 첫 출격 보정: 최소 생존권 1장 보장

목적:

- 플레이어가 첫 정산에서 아무것도 못 얻었다고 느끼지 않게 한다.
- 빠른 죽음 파밍을 막으면서도 첫 루프를 완성한다.

### 4.3 후보 배급권

후보 배급권은 런 행동으로 생성되고, 정산에서 승인/보류/손실 판정을 받는다.

예:

- 오픈하우스 거리 25초 체류: 신호표 후보 +1
- 로봇 적 15기 처리: 전원표 후보 +1
- 체력 35% 이하 12초 생존: 밥표 또는 전원표 후보 판정
- 반품 표식 적 20기 처치: 밥표 후보 승인 확률 상승

목적:

- 플레이어가 위험을 감수할 이유를 만든다.
- 카드 선택과 행동이 정산에 연결되게 한다.

## 5. 정산 결과 종류

정산 판정은 최소 다음 상태를 가진다.

| 결과 | 의미 | 플레이어 감각 |
| --- | --- | --- |
| 승인 | 후보가 실제 배급권으로 확정 | 해냈다 |
| 보류 | 후보가 이번에는 인정되지 않음 | 사유를 납득해야 함 |
| 오염 | 후보가 캠페인 압력/약관 영향으로 위험해짐 | 강하지만 찝찝함 |
| 손실 | 실패/약관 대가로 일부 후보 또는 확정권 약화 | 위험 선택의 결과 |
| 보존 | 실패했지만 일부 진행도 보존 | 다음 런 동기 |

### 5.1 승인

승인은 후보가 실제 배급권으로 확정되는 결과다.

문구 예:

```text
신호표 후보 1장 승인.
광고 시스템은 아직 이 실수를 모릅니다.
```

### 5.2 보류

보류는 후보가 사라지거나 다음 판정으로 넘어가는 결과다.

문구 예:

```text
전원표 후보 1장 보류.
로봇 고객님의 정상성이 충분히 증명되지 않았습니다.
```

주의:

- 보류는 억울한 랜덤으로 느껴지면 안 된다.
- 반드시 사유를 보여준다.

### 5.3 오염

오염은 후보가 승인되지만 대가가 붙는 결과다.

예:

- 신호표 승인, 캠페인 압력 +1
- 밥표 승인, 가족 적합성 검수 이벤트 가능성 증가
- 전원표 승인, 보안 로봇 등장 가능성 증가

문구 예:

```text
신호표가 승인되었습니다.
단, 해당 신호는 이미 광고 시스템에 한 번 읽혔습니다.
```

### 5.4 손실

손실은 위험한 약관 카드, 실패, 회수선 불안정으로 발생한다.

예:

- 확정 배급권 1장이 후보로 내려감
- 후보 승인 확률 감소
- 후보 일부 소실
- 캠페인 압력 상승

주의:

- 손실은 예고되어야 한다.
- 설명 없이 이미 얻은 것을 삭제하면 안 된다.

### 5.5 보존

보존은 실패했지만 일부 성과가 다음 런으로 이어지는 결과다.

예:

- 신호 위치 일부 보존
- 오픈하우스 체류 분석 +1
- 보스/결절 분석 일부 증가
- 보급소가 후보 일부를 임시 보관

목적:

- 실패가 완전 무의미해지지 않게 한다.
- 단, 실패 보상이 파밍 루프가 되면 안 된다.

## 8. 승인/보류 판정 원칙

### 8.1 판정은 완전 랜덤이 아니다

승인/보류는 확률을 가질 수 있다.

하지만 플레이어에게는 다음처럼 보이면 안 된다.

```text
운이 나빠서 못 받았다.
```

플레이어에게는 이렇게 보여야 한다.

```text
내 행동과 조건 때문에 이 후보는 승인/보류되었다.
```

### 8.2 승인 확률에 영향을 주는 요소

| 요소 | 영향 |
| --- | --- |
| 해당 후보 조건을 더 많이 수행 | 승인 확률 증가 |
| 관련 카드 보유 | 승인 확률 증가 |
| 위험 구역 체류 | 후보 생성 증가, 보류/오염 위험도 증가 |
| 캠페인 압력 높음 | 보류/오염/손실 위험 증가 |
| 약관 카드 사용 | 강한 보정, 대가 증가 |
| 안정적 귀환 | 승인 안정 |
| 일부 인양/검수 실패 | 손실/보류 위험 증가 |

### 8.3 표시 수준

0.2에서는 복잡한 확률 수치를 전부 보여주지 않는다.

표시는 다음 정도로 한다.

```text
승인 가능성: 높음 / 보통 / 낮음
보류 위험: 있음
오염 위험: 있음
```

또는 아이콘으로 표현한다.

## 9. 캠페인 압력

### 9.1 역할

캠페인 압력은 광고 환경이 윤서를 얼마나 정확히 인식하고 있는지를 나타낸다.

첫 5분에서는 세부 수치를 깊게 노출하지 않는다.

플레이어가 느껴야 하는 것:

> 욕심을 많이 냈거나 약관을 많이 썼기 때문에 광고가 나를 더 정확히 보고 있다.

### 9.2 상승 조건

캠페인 압력 상승 후보:

- 오픈하우스 거리 장기 체류
- 약관 카드 사용
- 정상 고객 스티커 만료
- 가족 할인 약관 사용
- 신호표 후보를 많이 생성
- 검문 로봇 다수 처리
- 회수선 불안정 상태에서 깊이 진입

### 9.3 영향

캠페인 압력 영향:

- 적 밀도 증가
- 검수 로봇 등장
- 배급권 후보 오염/보류 위험 증가
- 대신 신호표 후보 또는 고위험 보상 증가
- 다음 런 오픈하우스 구역 위험 증가

## 11. 보급소 배분

### 11.1 배분 목표

정산 후 플레이어는 배급권을 어디에 쓸지 결정한다.

0.2에서는 너무 복잡하게 하지 않는다.

### 11.2 최소 배분 선택

| 배분 | 즉시 효과 | 다음 런 변화 |
| --- | --- | --- |
| 밥표 -> 인간 구역 | 인간 안정도 +1 | 시작 회복/지원 후보 증가 |
| 전원표 -> 정비대/충전 스탠드 | 로봇 안정도/정비 기능 +1 | 도구/차징 카드 출현 보정 |
| 신호표 -> 출격 게시판 | 모델하우스 노출도 +1 | 오픈하우스 신호 위치 표시 |

### 11.3 배분 문구

밥표:

```text
인간 구역의 조명이 조금 늦게 꺼집니다.
```

전원표:

```text
정비대의 한쪽 팔이 다시 움직입니다.
```

신호표:

```text
출격 게시판에 새 핀이 꽂혔습니다.
```

### 11.4 부족 처리

부족이 즉시 게임오버가 되면 관리 게임처럼 피곤해진다.

0.2에서는 부족을 단계적 영향으로 둔다.

| 부족 자원 | 1차 영향 | 장기 영향 |
| --- | --- | --- |
| 밥표 부족 | 회복/인간 구역 분위기 약화 | 인간 안정도 이벤트 |
| 전원표 부족 | 정비대/로봇 지원 약화 | 로봇 동작 정지 이벤트 |
| 신호표 부족 | 지역 해석/보스 접근 지연 | 진행 속도 저하 |

## 17. 최종 요약

0.2 정산 시스템의 핵심은 다음이다.

> 밥표/전원표/신호표는 골드가 아니라 생존권이다.
> 플레이어 행동은 배급권 후보를 만든다.
> 정산은 그 후보를 승인, 보류, 오염, 손실, 보존으로 처리한다.
> 보급소 배분은 누가 하루 더 버티고 다음 출격이 어떻게 달라질지를 결정한다.

이 구조가 작동하면 `Atomic Ad Survivors`는 단순 전투 반복이 아니라, 전투 결과가 보급소와 지역 RPG로 이어지는 게임이 된다.

## 18. 1차 구현 반영 기록

### 18.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/main.gd`
- `scripts/run_result_evaluator.gd`

### 18.2 런 결과 입력

런 종료 시 `RunResultEvaluator`로 전달되는 값에 다음 항목이 추가됐다.

```text
card_counts
terms_failure_risk
open_house_time
audit_pass_count
audit_fail_count
audit_pressure_level
audit_total_processing
audit_last_result
```

이제 정산은 단순 생존 시간과 처치 수만 보지 않는다.

카드 선택, 약관 리스크, 오픈하우스 체류, 광고 감사 통과/미달, 총 처리량을 함께 보고 밥표/전원표/신호표 후보를 계산한다.

### 18.3 배급권 후보/승인 구조

`RunResultEvaluator`는 `ration_ticket_settlement`를 반환한다.

구조:

```text
ration_ticket_settlement
  confirmed
    food
    power
    signal
  candidates
    food
    power
    signal
  approved
    food
    power
    signal
  held
    food
    power
    signal
  downgraded
    food
    power
    signal
  approval_scores
  reason_lines
```

첫 구현 기준:

- 90초 이상 생존하면 밥표 1장이 확정된다.
- 120초 이상 생존하거나 `야근 정산표`를 보유하면 밥표 후보가 생긴다.
- 처리량/처치 판정 또는 `배터리 절약 영수증`이 전원표 후보를 만든다.
- 오픈하우스 25초/50초 체류와 `오픈하우스 방문 확인`이 신호표 후보를 만든다.
- 감사 통과는 승인 가능성을 올리고, 감사 미달/압력/약관 실패 리스크는 승인 가능성을 낮춘다.
- 약관 실패 리스크가 있는 상태로 실패/인양되면 확정 밥표가 후보로 내려갈 수 있다.

### 18.4 결과 화면 표시

결과 화면의 보상 라인에 다음 표시가 추가됐다.

```text
확정 배급권: 밥표 1
배급권 후보: 밥표 1, 신호표 1
후보 정산: 승인 신호표 1 / 보류 밥표 1
약관 리스크: 밥표 1 후보화
정산 사유: 오픈하우스 체류 25초: 신호표 후보 +1
```

정산 사유는 최대 3개까지만 먼저 보여준다.

목표는 상세 회계표가 아니라, 플레이어가 “내가 한 행동 때문에 이 결과가 나왔다”를 5초 안에 읽게 하는 것이다.

### 18.5 아직 남은 정산 과제

남은 과제:

1. 런 중 HUD에 현재 배급권 후보 수를 작게 표시한다.
2. 오염 판정을 별도 결과로 분리한다.
3. 후보/승인/보류 수치를 플레이테스트로 조정한다.
4. 카드별 처리량 기여도를 정산 사유에 더 정확히 연결한다.

## 19. 보급소 배분 1차 구현 기록

### 19.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/meta_progression.gd`
- `scripts/main.gd`
- `scripts/hud_controller.gd`

### 19.2 배급권 보관

정산에서 승인된 배급권은 보급소 보관량으로 들어간다.

반영 대상:

```text
확정 배급권
+ 후보 정산에서 승인된 배급권
= 보급소 보관 배급권
```

보관 자원:

```text
밥표
전원표
신호표
```

결과 화면에는 다음처럼 추가 표시된다.

```text
보급소 보관: 밥표 +1, 신호표 +1
```

### 19.3 배분 선택지

보급소 화면의 정산 카운터에 배급권 배분 선택지가 추가됐다.

| 선택지 | 비용 | 다음 출격 변화 |
| --- | --- | --- |
| 밥표 -> 인간 구역 | 밥표 1 | 최대 HP +8 |
| 전원표 -> 정비대 | 전원표 1 | 차징 피해 +1, 차징 쿨다운 -0.15초, 차징 입력 윈도우 +0.04초 |
| 신호표 -> 출격 게시판 | 신호표 1 | 오픈하우스 신호 체류 기준 완화 |

이 수치는 0.2 체감용 1차 값이다.

최종 밸런스가 아니라 “배분했더니 다음 출격이 달라졌다”를 먼저 확인하기 위한 값이다.

### 19.4 다음 런 변화

배분은 `MetaProgression.bonuses()`에 반영된다.

현재 반영:

- 인간 구역 배분: 최대 HP 보정
- 정비대 배분: 차징 피해/쿨다운/입력 윈도우 보정
- 출격 게시판 배분: 오픈하우스 신호 후보 발생 기준 완화

보급소 화면에는 다음 변화 요약을 표시한다.

```text
다음 변화: 인간 구역 지원으로 최대 HP +8 / 정비대 전원으로 차징 0.15초 단축
```

### 19.5 아직 남은 보급소 과제

남은 과제:

1. 보급소 배분 결과를 인간/로봇/NPC 반응 문구로 확장한다.
2. 배분 효과를 영구 누적과 다음 런 1회성 효과로 분리할지 결정한다.
3. 런 중 HUD에 현재 후보 배급권을 표시한다.
4. 신호표 배분이 지도/오픈하우스 방향 표시로 더 직접 보이게 한다.
5. 플레이테스트로 배분 보너스 수치를 조정한다.

## 20. 런 중 후보 HUD/오염 정산 1차 구현 기록

### 20.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/main.gd`
- `scripts/run_result_evaluator.gd`
- `scripts/hud_controller.gd`

### 20.2 런 중 배급권 후보 HUD

런 중 HUD에 현재 정산 전망이 작게 표시된다.

표시 형식:

```text
정산 밥1+1 전1 신2  오염
```

의미:

- `밥1+1`: 확정 밥표 1장 + 밥표 후보 1장
- `전1`: 전원표 후보 1장
- `신2`: 신호표 후보 2장
- `오염/약관/주의`: 현재 정산 리스크

목표:

> 플레이어가 정산 화면에 도착하기 전에 이미 “내 행동이 배급권 후보로 쌓이고 있다”를 보게 한다.

### 20.3 라이브 후보 계산

현재 라이브 HUD는 정산 결과를 완전히 확정하지 않고, 런 중 조건을 기준으로 후보 전망을 계산한다.

반영 조건:

- 90초 이상 생존: 확정 밥표 전망
- 생존 시간/야근 정산표/저체력 조건: 밥표 후보 전망
- 로봇 처리/배터리 절약 영수증/로봇 명령 뒤집기: 전원표 후보 전망
- 오픈하우스 신호 단계/오픈하우스 방문 확인/배수로 체류: 신호표 후보 전망
- 약관 리스크/감사 압력/가짜 귀환로 체류: 위험 표시

런 중 HUD는 플레이어 안내용이고, 최종 승인/보류/오염은 `RunResultEvaluator`에서 정산한다.

### 20.4 오염 정산

`ration_ticket_settlement`에 `contaminated`와 `contamination_lines`가 추가됐다.

오염은 후보가 승인되었지만 광고 환경의 꼬리표가 붙은 상태다.

현재 오염 조건:

- 감사 압력 3 이상이고 승인 안정성이 낮음
- 약관 실패 리스크가 높음
- 오픈하우스 신호가 깊게 진행된 상태에서 신호표 승인
- 끊긴 광고 산책로에서 밥표 후보를 만든 상태에서 압력 존재

결과 화면 예:

```text
오염 정산: 신호표 1
오염 사유: 신호표 승인에 광고 오염 꼬리표가 붙었습니다.
```

### 20.5 지역 약관 차별화

R01의 보조 루트가 정산에 서로 다르게 반영된다.

| 구역 | 정산 역할 | 리스크 |
| --- | --- | --- |
| 배수로 침묵 주머니 | 신호표 후보/승인 보정 | 조용한 우회, 압력 직접 증가는 낮음 |
| 끊긴 광고 산책로 | 밥표 후보 | 오염 위험, 가짜 귀환 감각 |

결과 화면 예:

```text
지역 약관: 배수로 침묵 주머니 24초 - 신호표 후보가 조용히 올라왔습니다.
지역 약관: 끊긴 광고 산책로 22초 - 밥표 후보와 오염 위험이 함께 생겼습니다.
```

### 20.6 닫힌 과제와 남은 과제

닫힌 과제:

- 런 중 HUD에 현재 후보 배급권 표시.
- 오염 판정을 별도 결과로 분리.
- 오픈하우스/배수로/끊긴 광고 산책로의 지역 약관 차별화 1차 적용.

남은 과제:

1. 카드별 처리량 기여도를 정산 화면에 더 정확히 표시한다.
2. 보급소 배분 결과를 인간/로봇/NPC 반응 문구로 확장한다.
3. 신호표 배분이 지도/오픈하우스 방향 표시로 더 직접 보이게 한다.
4. 실제 플레이테스트로 후보/승인/보류/오염 수치를 조정한다.

## 21. 카드별 정산 기여 1차 구현 기록

### 21.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/main.gd`
- `scripts/run_result_evaluator.gd`

### 21.2 목적

정산 화면에서 플레이어가 다음을 이해하게 한다.

> 내가 고른 카드가 이번 런의 처리량과 배급권 정산에 실제로 기여했다.

기존에는 정산 사유가 구역/감사/후보 중심이었다.

이제 결과 화면에 카드 단위 기여도도 표시된다.

### 21.3 표시 예

```text
카드 기여: 반품 라벨 처리량 +380 (반품 표식)
카드 기여: 송출 잔류 처리량 +260 (송출 잔류)
카드 기여: 배터리 절약 영수증 - 전원표 후보/승인 보정
```

### 21.4 기록 방식

런 중 `main.gd`가 카드별 기여 로그를 쌓는다.

기여 로그:

```text
card_id
name
processing
events
notes
```

런 종료 시 `RunResultEvaluator`가 이 로그를 받아 처리량 상위 카드를 결과 라인으로 변환한다.

### 21.5 정산 전용 카드 보정

처리량으로 크게 드러나지 않지만 정산에 영향을 준 카드도 보조 표시된다.

예:

- 야근 정산표: 밥표 후보와 감사 기준 완화
- 배터리 절약 영수증: 전원표 후보/승인 보정
- 오픈하우스 방문 확인: 신호표 후보와 위험 구역 처리량
- 반품 불가 동의서: 차징 처리량과 실패 정산 위험

### 21.6 닫힌 과제와 남은 과제

닫힌 과제:

- 카드별 처리량 기여도 정산 표시 1차 구현.

남은 과제:

1. 카드 기여 산식이 플레이어 체감과 맞는지 플레이테스트로 확인한다.
2. 카드 기여를 별도 상세 정산 패널로 확장할지 결정한다.
3. 오염 결과와 카드 기여를 다음 런 지역 변화로 연결한다.

## 22. 보급소 배분 반응 1차 구현 기록

### 22.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/meta_progression.gd`
- `scripts/main.gd`
- `scripts/hud_controller.gd`

### 22.2 목적

보급소 배분이 단순 수치 강화처럼 보이지 않게 한다.

플레이어가 느껴야 하는 것:

> 내가 방금 밥표/전원표/신호표를 어디에 줬고, 그 선택 때문에 보급소 안의 누군가 또는 어떤 시설이 반응했다.

### 22.3 적용된 반응

| 배분 | 즉시 반응 | 누적 반응 방향 |
| --- | --- | --- |
| 밥표 -> 인간 구역 | 인간 구역에서 누군가 빈 그릇을 내려놓음 | 조명이 더 오래 켜짐, 미나 반응 |
| 전원표 -> 정비대 | 정비대의 한쪽 팔이 낮은 소리로 돌아감 | 팝시/정비대/차징 접점 반응 |
| 신호표 -> 출격 게시판 | 세븐이 신호표를 게시판에 고정 | 게시판이 오픈하우스 방향을 더 정확히 표시 |

### 22.4 UI 표시

보급소 화면에는 세 층의 반응이 표시된다.

1. 현재 배분 상태 요약
2. 누적 보급소 반응 요약
3. 방금 적용한 배분의 즉시 반응

예:

```text
반응: 인간 구역의 조명이 조금 늦게 꺼집니다.
적용 완료: 밥표 -> 인간 구역 - 인간 구역에서 누군가 빈 그릇을 내려놓았습니다.
```

선택지에도 반응 예고가 붙는다.

```text
밥표 -> 인간 구역
다음 출격 최대 HP +8 / 인간 구역 안정 +1
반응: 인간 구역의 조명이 조금 늦게 꺼집니다.
```

### 22.5 의미

이제 보급소 배분은 다음 구조를 가진다.

```text
배급권 획득
-> 보급소 보관
-> 배분 선택
-> 시설/NPC 반응
-> 다음 런 수치 변화
```

수치와 감정 피드백이 함께 움직이므로, 보급소가 상점이 아니라 살아남는 장소로 읽힐 가능성이 커졌다.

### 22.6 닫힌 과제와 남은 과제

닫힌 과제:

- 보급소 배분에 인간/로봇/NPC/시설 반응 문구 1차 연결.

남은 과제:

1. 신호표 배분이 지도/오픈하우스 방향 표시로 더 직접 보이게 한다.
2. 오염 결과를 다음 런 지역 변화나 노선 기록으로 연결한다.
3. 보급소 반응을 실제 NPC 대사/이벤트 로그로 확장할지 결정한다.

## 23. 신호표 배분 지도 반영 1차 구현 기록

### 23.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/meta_progression.gd`
- `scripts/main.gd`
- `scripts/hud_controller.gd`

### 23.2 목적

신호표를 출격 게시판에 배분했을 때, 다음 출격이 실제로 달라졌다는 감각을 화면에서 보여준다.

기존에는 신호표 배분이 `오픈하우스 신호 기준 완화`라는 수치 변화로만 보였다.
이번 구현은 그 효과를 다음처럼 시각화한다.

```text
신호표 -> 출격 게시판
-> 게시판 문구 변화
-> 런 시작 안내 문구 변화
-> 월드 지도에 오픈하우스/모델하우스 방향 핀 표시
-> 플레이어 주변 나침반 화살표 표시
```

### 23.3 배분 단계별 표시

| 신호표 배분 누적 | 표시 변화 |
| --- | --- |
| 0 | 게시판이 모델하우스 방향을 아직 붙잡지 못함 |
| 1 | 오픈하우스 거리 방향 핀 표시 |
| 2 | 오픈하우스에서 모델하우스로 이어지는 경로선 표시 |
| 3 이상 | 모델하우스 결절 핀이 먼저 깜박임 |

### 23.4 런 중 표시

런 중에는 다음이 표시된다.

- 플레이어 주변 방향 나침반
- 오픈하우스 신호표 후보 핀
- 신호표 2장 이상 배분 시 모델하우스 신호 핀
- 오픈하우스 접근 후에는 목표 화살표가 모델하우스 쪽으로 전환

이 표시의 목적은 자동 네비게이션이 아니다.
목적은 플레이어가 “신호표를 쓴 덕분에 다음 목표가 화면에 붙잡혔다”고 느끼게 하는 것이다.

### 23.5 닫힌 과제와 남은 과제

닫힌 과제:

- 신호표 배분이 지도/오픈하우스 방향 표시로 보이는 1차 연결.

남은 과제:

1. 표시가 너무 친절해서 탐험감을 죽이지 않는지 확인한다.
2. 지역 약관 예고와 신호표 방향 표시가 서로 충돌하지 않게 정리한다.
3. 오염된 신호표가 잘못된 방향 표시를 만들 수 있는지 후속 검토한다.

## 24. 지역 약관 예고 1차 구현 기록

### 24.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/main.gd`
- `scripts/hud_controller.gd`

### 24.2 목적

지역 약관은 정산 후에만 알게 되면 “랜덤 보상표”처럼 느껴질 수 있다.

따라서 이번 구현은 플레이어가 출격 전과 런 중에 미리 약관을 읽고, 그 위험을 감수할지 판단하게 만드는 것을 목표로 한다.

```text
출격 전 약관 예고

```

### 24.3 표시 위치

| 위치 | 표시 내용 |
| --- | --- |
| 보급소 출격 게시판 | 다음 출격의 대표 지역 약관 |

| 구역 진입 배너 | 해당 구역의 약관 한 줄 |
| 런 중 정산 HUD | 현재 구역의 짧은 약관 태그 |

### 24.4 예고되는 약관

| 구역 | 약관 예고 |
| --- | --- |

### 24.5 의미

이제 배급권 정산은 다음처럼 읽힌다.

```text
플레이어가 약관을 먼저 봄
-> 위험 구역에 들어감
-> 조건을 채움
```

정산 결과가 뒤늦은 설명이 아니라, 미리 예고된 규칙의 결과로 보일 가능성이 커졌다.

닫힌 과제:

- 지역 약관을 런 시작/출격 게시판/구역 진입/HUD에 미리 예고하는 1차 연결.

남은 과제:

1. 약관 예고 문구가 모바일 화면에서 너무 길지 않은지 확인한다.
2. 오염 표식이 전투 가독성을 해치지 않는지 확인한다.
3. 오염된 신호표가 잘못된 지도 표시를 만들지 여부를 후속 검토한다.

## 25. 오염 결과 다음 런 반영 1차 구현 기록

### 25.1 구현 반영 파일

이번 구현으로 반영된 파일:

- `scripts/local_response_state.gd`
- `scripts/meta_progression.gd`
- `scripts/main.gd`
- `scripts/route_phrase_resolver.gd`

### 25.2 목적

오염된 배급권은 정산 화면에서만 보이고 사라지면 안 된다.

플레이어가 오염 후보를 승인하거나 끌고 돌아온 뒤에는, 다음 출격에서 R01의 감사 압력과 노선 문구, 맵 흔적이 바뀌어야 한다.

```text
오염 후보 발생
-> 정산에서 오염 기록 누적
-> 보급소/출격 게시판에 다음 변화 표시
-> 다음 런 약관과 적 압력 변화
-> 맵 위 오염 흔적 표시
```

### 25.3 누적되는 오염 상태

R01 지역 상태에 다음 값이 누적된다.

| 값 | 의미 |
| --- | --- |
| `r01_contaminated_food_count` | 오염 밥표 누적 |
| `r01_contaminated_power_count` | 오염 전원표 누적 |
| `r01_contaminated_signal_count` | 오염 신호표 누적 |
| `r01_contamination_total` | 전체 오염 꼬리표 누적 |
| `r01_last_contamination_ticket` | 마지막으로 강하게 남은 오염 종류 |

### 25.4 다음 런 반영

오염이 누적되면 다음 런에 다음 변화가 생긴다.

| 오염 종류 | 다음 런 변화 |
| --- | --- |
| 오염 밥표 | 끊긴 광고 산책로 감사와 오염 위험 강조 |
| 오염 전원표 | 로봇 검수/장갑 광고 비중 증가 |
| 오염 신호표 | 오픈하우스/모델하우스 신호 감사 강화 |

전투 압력도 소폭 오른다.

- 스폰 압력 증가
- 적 이동 속도 소폭 증가
- 엘리트 출현 판정 소폭 증가
- 오염 종류에 따라 특정 적 역할 가중치 변화

### 25.5 화면 표시

오염 결과는 다음 위치에서 보인다.

- 정산 후 보상/진행 줄
- 보급소 출격 게시판 노선 문구
- 보급소 하단 지역 약관 예고
- 런 시작 약관 공지
- 런 중 정산 HUD 약관 태그
- 맵 위 오염 표식

맵 표식은 다음 앵커에 붙는다.

| 오염 종류 | 표시 위치 |
| --- | --- |
| 오염 밥표 | 끊긴 광고 산책로 |
| 오염 전원표 | 분양 주택 루프 |
| 오염 신호표 | 오픈하우스 거리, 누적 시 모델하우스 접근로 |

### 25.6 의미

이제 오염은 단순 실패 문구가 아니라 지역 상태다.

플레이어는 다음처럼 읽을 수 있다.

```text
내가 오염된 표를 들고 돌아왔다.
-> 보급소는 그걸 기록했다.
-> 다음 출격에서 지역 감사가 나를 더 정확히 본다.
```

검증할 질문:

1. 플레이어가 오염 결과를 다음 출격 변화로 인식하는가?
2. 오염 압력이 불합리한 처벌이 아니라 선택의 뒤끝으로 느껴지는가?
3. 오염 표식이 전투 가독성을 방해하지 않는가?

## 26. 정산과 보급소 NPC 로그 연결 1차 구현 기록

### 26.1 목적

정산이 숫자와 보상표로만 보이면 배급권이 골드처럼 느껴진다.

이번 구현은 배급권 정산 결과를 보급소 NPC 반응으로 남겨, 플레이어가 “내가 가져온 표가 누군가의 오늘을 바꿨다”고 느끼게 하는 것을 목표로 한다.

### 26.2 로그가 생기는 조건

| 조건 | 대표 NPC | 로그 방향 |
| --- | --- | --- |
| 첫 회수 | 미나 | 윤서의 이름과 귀환 확인 |
| 배급권 정산 | 미나 | 보급소 이름으로 보관 |
| 오염 밥표 | 미나 | 먹을 수 있지만 뒤끝이 남음 |
| 오염 전원표 | 팝시 | 검수음/전원표 불안 |
| 오염 신호표 | 세븐 | 지우지 않고 다음 기준으로 고정 |
| 신호 단서 | 세븐 | 잡음/지도/좌표 기록 |
| 보스 분석 | 도윤 | 다음 대응 가능성 |
| 보스 결과 | 도윤/복희/세븐 | 결절음, 기억, 처리 기록 |
| 보급소 배분 | 미나/팝시/세븐 | 각 시설 반응 |
| 정비 강화 | 도윤 | 출격 전 점검표 반영 |

### 26.3 UI 표시

보급소 UI에는 최근 이벤트가 다음 형태로 표시된다.

```text
보급소 로그: 세븐: 오염 신호는 지우지 않습니다. 다음 출격 기준으로 둡니다.
```

이 로그는 전체 대화창이 아니다.
정산과 배분을 감정적으로 닫는 짧은 증거다.

### 26.4 검증할 질문

1. 정산 결과가 NPC 반응으로 이어져 생존권처럼 느껴지는가?
2. 오염 결과가 NPC별 관점 차이로 보이는가?
3. 같은 보급소 화면 안에서 로그, 배분 버튼, 재출격 목표가 서로 충돌하지 않는가?
